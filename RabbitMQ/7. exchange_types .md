RabbitMQ exchange types define **how messages are routed from an exchange to queues**.

An exchange never stores messages—it only decides **where they should go**.

The exchange type determines the routing logic.

---

# Core Idea

```text id="core001"
Producer -> Exchange -> Queue(s)
```

The exchange inspects message metadata and applies rules.

Those rules differ by type.

---

# 1) Direct Exchange

A **direct exchange** routes messages using an **exact match** between:

* message routing key
* queue binding key

---

## How It Works

If producer sends:

```text id="d1"
routing_key = send_otp
```

And queue is bound with:

```text id="d2"
binding_key = send_otp
```

Then it matches.

---

If binding is:

```text id="d3"
send_email
```

Then it does not match.

---

---

## Diagram

```text id="d4"
          [Direct Exchange]
             /        \
   send_otp /          \ send_email
           /            \
 [transactional.queue] [notification.queue]
```

---

---

## Use Cases

* task queues
* job dispatch
* command execution
* specific service routing

---

---

## Advantages

* simple
* fast
* deterministic

---

---

## Limitation

No pattern matching.

Only exact equality.

---

---

# 2) Topic Exchange

A **topic exchange** routes using **pattern matching**.

Routing keys are dot-separated words.

---

Example:

```text id="t1"
campaign.created
payment.completed
user.signup
```

---

Binding keys may contain wildcards.

---

## Wildcards

---

### `*`

Matches exactly one word.

---

Example:

```text id="t2"
campaign.*
```

Matches:

```text id="t3"
campaign.created
campaign.updated
```

But not:

```text id="t4"
campaign.email.sent
```

---

---

### `#`

Matches zero or more words.

---

Example:

```text id="t5"
campaign.#
```

Matches:

```text id="t6"
campaign.created
campaign.email.sent
campaign
```

---

---

## Diagram

```text id="t7"
              [Topic Exchange]
                 /         \
        campaign.*       *.completed
             |                |
   [campaign.queue]    [audit.queue]
```

---

---

## Use Cases

* event-driven systems
* microservices
* logging categories
* flexible subscriptions

---

---

## Advantages

* scalable
* expressive
* supports multi-consumer routing

---

---

## Limitation

Slightly more complex than direct.

---

---

# 3) Fanout Exchange

A **fanout exchange** ignores routing keys.

It sends every message to **all bound queues**.

---

---

## How It Works

Producer sends:

```text id="f1"
routing_key = anything
```

Exchange ignores it.

Broadcasts to all.

---

---

## Diagram

```text id="f2"
             [Fanout Exchange]
            /        |        \
           /         |         \
 [queue A]     [queue B]     [queue C]
```

---

Every queue gets a copy.

---

---

## Use Cases

* notifications
* pub/sub
* real-time dashboards
* cache invalidation

---

---

## Advantages

* simple broadcasting
* decouples subscribers

---

---

## Limitation

No filtering.

Everyone receives everything.

---

---

# 4) Headers Exchange

A **headers exchange** routes based on message headers instead of routing key.

---

---

## How It Works

Producer sends headers:

```text id="h1"
department = billing
priority = high
```

Queue binding defines required headers.

---

Example:

```text id="h2"
x-match = all
department = billing
priority = high
```

---

Only messages matching rules are routed.

---

---

## Matching Modes

---

### `x-match = all`

All headers must match.

---

---

### `x-match = any`

At least one header must match.

---

---

## Diagram

```text id="h3"
         [Headers Exchange]
                |
     checks metadata fields
                |
         [billing.queue]
```

---

---

## Use Cases

* complex filtering
* metadata-based routing
* enterprise integrations

---

---

## Advantages

* powerful routing rules

---

---

## Limitation

* slower
* harder to manage
* less commonly used

---

---

# Comparison Table

| Exchange Type | Routing Logic  | Uses Routing Key? | Pattern Support |
| ------------- | -------------- | ----------------- | --------------- |
| Direct        | exact match    | yes               | no              |
| Topic         | wildcard match | yes               | yes             |
| Fanout        | broadcast      | no                | no              |
| Headers       | header match   | no                | header-based    |

# Internal Broker Perspective

When message arrives:

---

## Direct

```text id="i1"
if routing_key == binding_key
```

---

---

## Topic

```text id="i2"
if wildcard_pattern_matches(routing_key)
```

---

---

## Fanout

```text id="i3"
deliver_to_all_bound_queues()
```

---

---

## Headers

```text id="i4"
if headers_match(binding_rules)
```

---

---

# Which One Should You Choose?

---

## Direct → commands/tasks

---

## Topic → events/categories

---

## Fanout → broadcasting

---

## Headers → specialized metadata routing

---

---

# Production Advice

Most real systems rely mainly on:

* direct
* topic

Fanout for pub/sub.

Headers only when truly needed.

---

# Final Understanding

Exchange type is the routing strategy.

It defines the decision-making algorithm between producer and queue.

That is its architectural role.
