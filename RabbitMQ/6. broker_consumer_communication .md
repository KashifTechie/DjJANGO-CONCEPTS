# Big Picture

RabbitMQ is a **broker-managed push-based delivery system**.

That means:

* producer sends message to broker
* broker stores message in queue
* broker pushes message to consumer
* consumer sends ACK back

So **consumer does not pull messages manually** in normal RabbitMQ usage.

Broker delivers them.

---

# Full Lifecycle

---

## Stage 1 — Producer Publishes Message

```python
channel.basic_publish(
    exchange="jobs.direct",
    routing_key="send_otp",
    body='{"user": 42}'
)
```

---

## Broker Receives It

Broker evaluates exchange bindings.

Finds matching queue.

Stores message.

---

---

# Internal Queue Storage

Imagine queue as two logical buckets:

```text
QUEUE: transactional.queue

+-------------------+
| READY             |
|-------------------|
| Msg#101           |
| Msg#102           |
| Msg#103           |
+-------------------+

| UNACKED           |
|-------------------|
| (empty)           |
+-------------------+
```

---

`READY` = waiting for delivery
`UNACKED` = delivered but not confirmed

---

---

# Stage 2 — Consumer Subscribes

Consumer runs:

```python
channel.basic_consume(
    queue="transactional.queue",
    on_message_callback=callback
)
channel.start_consuming()
```

---

This does **not** fetch one message.

It registers interest.

Like saying:

> I am ready. Send messages to me.

---

So consumer opens a persistent TCP connection.

Broker now knows:

```text
Consumer C1 attached to transactional.queue
```

---

---

# Stage 3 — Broker Pushes Message

Broker checks available consumers.

Picks one.

Moves message from READY → UNACKED.

---

```text
Before Delivery:

READY:   [101,102,103]
UNACKED: []

After Sending Msg#101:

READY:   [102,103]
UNACKED: [101]
```

---

Then sends packet over TCP.

---

---

# What Consumer Receives

Callback gets:

```python
def callback(ch, method, properties, body):
```

---

## Internals

---

### `body`

```python
b'{"user":42}'
```

Actual payload.

---

---

### `method`

Contains delivery metadata.

Example:

```python
method.delivery_tag = 17
method.routing_key = "send_otp"
```

---

---

### `properties`

Headers, correlation IDs, expiration, etc.

---

---

# Stage 4 — Consumer Processes

```python
process_message(body)
```

This is your logic.

---

---

# Stage 5 — Consumer ACKs

```python
ch.basic_ack(delivery_tag=method.delivery_tag)
```

---

This sends protocol frame back to broker.

---

---

# Broker Updates Queue State

Removes message from UNACKED.

---

```text
Before ACK:

READY:   [102,103]
UNACKED: [101]

After ACK:

READY:   [102,103]
UNACKED: []
```

---

Message is permanently deleted.

# Is It Push or Pull?

---

## RabbitMQ = Push

Broker sends messages proactively.

---

Consumer subscribes once.

Then broker streams deliveries.

---

This differs from Kafka.

Kafka consumers poll.

RabbitMQ consumers receive pushed messages.

---

---

# What If Consumer Crashes?

Suppose:

---

```text
READY:   [102,103]
UNACKED: [101]
```

---

Before ACK, consumer disconnects.

Broker detects channel close.

---

Moves message back:

```text
READY:   [101,102,103]
UNACKED: []
```

---

So another consumer can process it.

---

---

# Why UNACKED Exists

To protect against message loss.

It is temporary holding state.

---

---

# How Broker Knows Which Consumer Gets Message

Uses:

* round-robin dispatch
* respects `prefetch_count`

---

If:

```python
channel.basic_qos(prefetch_count=1)
```

Broker sends only one unacked message at a time.

---

So overloaded consumers are protected.

# What ACK Actually Is

Not a function call inside broker.

It is a protocol command over AMQP.

---

Consumer sends:

```text
Basic.Ack(delivery_tag=17)
```

Broker matches it to stored delivery.

Deletes tracked message.

---

---

# Final Mental Model

```text
READY   = waiting shelf
UNACKED = borrowed but not returned
ACK     = confirmed completion
NACK    = failed return
```

---

RabbitMQ is essentially tracking loans.

Not just storage.

---

# Key Insight

The queue is not a simple list.

It is a **state machine**:

```text
Stored -> Delivered -> Awaiting Ack -> Removed / Requeued
```

That is why RabbitMQ is reliable.

---

That internal transition system is the real engine.
