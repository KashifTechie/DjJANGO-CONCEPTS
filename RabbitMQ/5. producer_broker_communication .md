Good—now we trace the **other half** of the system:

**Producer → Exchange → Queue storage**

This is where many people oversimplify RabbitMQ as “send to queue.”

That’s inaccurate.

A producer usually **never sends directly to a queue**.

It sends to an **exchange**, and the broker decides where to store the message.

---

# High-Level Pipeline

```text
Producer -> Broker Connection -> Exchange -> Routing Logic -> Queue -> Stored as READY
```

---

# What Actually Happens Internally

We’ll follow one message:

```json
{"user_id": 42, "action": "otp"}
```

Routing key:

```text
send_otp
```

---

# Stage 1 — Producer Opens Connection

Python code:

```python
connection = pika.BlockingConnection(params)
channel = connection.channel()
```

---

## Internally

A TCP socket is created.

Then AMQP handshake occurs.

---

```text
Producer ---------------------- Broker
          TCP SYN
          TCP ACK
          AMQP Start
          AMQP Tune
          AMQP Open
```

---

Now a logical **channel** exists.

A channel is a lightweight virtual session over one TCP connection.

---

---

# Stage 2 — Producer Publishes Message

```python
channel.basic_publish(
    exchange="jobs.direct",
    routing_key="send_otp",
    body='{"user_id":42}'
)
```

---

This does **not** mean “put in queue.”

It means:

> Send this message to exchange `jobs.direct` with routing label `send_otp`.

---

---

# What Broker Receives

AMQP frame contains:

---

## Metadata

```text
Exchange = jobs.direct
Routing Key = send_otp
Properties = delivery_mode=2
```

---

## Payload

```text
{"user_id":42}
```

---

---

# Internal Representation

```text
Incoming Publish Frame
+-----------------------------------+
| exchange: jobs.direct             |
| routing_key: send_otp             |
| persistent: true                  |
| body: {"user_id":42}              |
+-----------------------------------+
```

---

---

# Stage 3 — Exchange Lookup

Broker checks:

```text
Does exchange 'jobs.direct' exist?
```

If not → error.

If yes → continue.

---

---

# Exchange Object

```text
Exchange: jobs.direct
Type: direct
Bindings:
   send_otp   -> transactional.queue
   send_email -> notification.queue
```

---

Exchange does **not** store messages.

It only routes.

Think of it as a traffic controller.

---

---

# Stage 4 — Routing Decision

Because exchange type = **direct**:

Rule is exact match.

---

Broker compares:

```text
Message Routing Key = send_otp
Binding Key         = send_otp
```

Match found.

---

---

# Stage 5 — Queue Accepts Message

Broker writes message into queue structure.

---

Before insert:

```text
transactional.queue

READY:   [101,102]
UNACKED: []
```

---

After insert:

```text
transactional.queue

READY:   [101,102,103]
UNACKED: []
```

---

New message becomes READY.

---

---

# Internal Storage Model

```text
Queue Record
+----------------------------------+
| msg_id: 103                      |
| payload: {"user_id":42}          |
| persistent: true                 |
| timestamp: 21:44:01              |
| state: READY                     |
+----------------------------------+
```

---

---

# If Persistent Message

If:

```python
properties=pika.BasicProperties(delivery_mode=2)
```

Then broker also writes to disk.

---

---

## Disk + Memory

```text
Memory Queue Index -> Message Metadata
Disk Segment       -> Full Persistent Payload
```

---

So restart does not lose message.

---

---

# Full Internal Flow Diagram

```text
+-------------+
| Producer    |
+-------------+
      |
      | basic_publish(exchange, routing_key, body)
      v
+----------------------+
| TCP + AMQP Channel   |
+----------------------+
      |
      v
+----------------------+
| Broker Receive Frame |
+----------------------+
      |
      v
+----------------------+
| Exchange Lookup      |
| jobs.direct          |
+----------------------+
      |
      | direct match
      v
+----------------------+
| Binding Resolution   |
| send_otp -> queue    |
+----------------------+
      |
      v
+----------------------+
| transactional.queue  |
| READY += message     |
+----------------------+
```

---

---

# What If No Matching Binding?

Message is unroutable.

---

## Without Mandatory Flag

Silently dropped.

---

## With Mandatory=True

Returned to producer.

---

Example:

```python
channel.basic_publish(..., mandatory=True)
```

---

Then producer gets notification.

---

---

# What If Multiple Bindings Match?

For topic/fanout:

message copied to all matching queues.

---

Example:

```text
events.topic
   campaign.* -> campaign.queue
   *.created  -> audit.queue
```

---

Routing key:

```text
campaign.created
```

---

Stored in:

```text
campaign.queue
audit.queue
```

---

One publish → multiple queue inserts.

---

---

# Queue State After Storage

```text
transactional.queue

READY:
  Msg#101
  Msg#102
  Msg#103   <- newly stored

UNACKED:
  none
```

---

No consumer involved yet.

This is purely broker-side.

---

---

# Important Insight

Exchanges are stateless routers.

Queues are stateful storage.

---

---

# Producer Never Knows Queue

Producer only knows:

```text
exchange + routing_key
```

---

Broker decides destination.

This decouples producer from consumers.

---

---

# Mental Model

```text
Producer = sender
Exchange = post office sorter
Queue = mailbox
Consumer = receiver
```

---

Sender writes address label.

Sorter decides mailbox.

Mailbox stores until collected.

---

---

# End-to-End Lifecycle (Until Stored)

```text
Create TCP connection
      ↓
Open AMQP channel
      ↓
Publish frame sent
      ↓
Broker parses metadata
      ↓
Exchange selected
      ↓
Bindings checked
      ↓
Matching queue found
      ↓
Message appended to READY
      ↓
(Optional) persisted to disk
```

---

---

# Final Truth

RabbitMQ message storage is **not direct enqueueing**.

It is a routing pipeline with broker-controlled delivery.

That architecture is what gives RabbitMQ flexibility and reliability.
