# RabbitMQ in Python — Production-Level Setup for Complex Projects

Now we move from theory to **real architecture for large systems**.

You asked the right question:

> If producer needs exchange, queue, routing key — how do we manage this in a complex project with many services?

That is exactly where production design matters.

Because in real systems, you do **not** hardcode everything randomly.

You design a **messaging topology**.

---

# Real-World Scenario

Suppose your platform has these domains:

* **Campaign Service**
* **Transactional Messaging**
* **Contact Importing**
* **Workflow Execution**
* **Payment Processing**

Each has different responsibilities.

If all use one queue, chaos.

So we structure them properly.

# 2) Can We Have Multiple Exchanges and Multiple Queues?

Yes.

And in production, you often should.

RabbitMQ fully supports:

* multiple exchanges
* multiple queues
* multiple bindings
* multiple routing keys

---

---

# Why?

Because each domain has different traffic patterns.

Separating them improves:

* maintainability
* observability
* scaling
* failure isolation

# 4) How Routing Key Helps

Routing key tells exchange how to route message.

---

---

## Example

Producer sends:

```text id="rk001"
exchange = events.topic
routing_key = payment.success
```

---

Topic exchange checks bindings:

```text id="rk002"
payment.*
workflow.*
campaign.*
```

---

Message goes to queues matching `payment.*`

---

---

## Why Powerful?

One exchange can route to many services.

This avoids hardcoding destination queues.

# 8) Connection Management

## `connection.py`

```python
from pika import ConnectionParameters, BlockingConnection, PlainCredentials
from django.conf import settings

def get_channel():
    
    credentials = PlainCredentials(
        settings.RMQ_USERNAME,
        settings.RMQ_PASSWORD,
    )

    parameters = ConnectionParameters(
            host=settings.RMQ_HOST,
            port= settings.RMQ_PORT,
            credentials=credentials,
            heartbeat=60,
            blocked_connection_timeout=30
    )   
    
    connection = BlockingConnection(parameters)

    channel = connection.channel()

    return connection, channel
```

---

---

# Production Note

In real systems:

* reuse connections
* create channels per thread/task
* avoid reconnecting every publish

# 9) Dynamic Topology Setup

## `router.py`

```python
from messaging.config import RABBITMQ_TOPOLOGY
from messaging.connection import get_channel

def setup_topology():
    channel = get_channel()

    for exchange in RABBITMQ_TOPOLOGY["exchanges"].values():
        channel.exchange_declare(
            exchange=exchange["name"],
            exchange_type=exchange["type"],
            durable=True
        )

    for queue in RABBITMQ_TOPOLOGY["queues"].values():
        channel.queue_declare(queue=queue["name"], durable=True)

        exchange_name, routing_key = queue["binding"]

        channel.queue_bind(
            exchange=exchange_name,
            queue=queue["name"],
            routing_key=routing_key
        )
```

---

---

# This Enables Dynamic Infrastructure Creation

Broker topology becomes code-defined.

Infrastructure as configuration.

# 12) Consumer Side

## `consumer.py`

```python
def consume(queue_name, handler):
    channel = get_channel()

    channel.basic_qos(prefetch_count=10)

    def callback(ch, method, properties, body):
        handler(body)
        ch.basic_ack(method.delivery_tag)

    channel.basic_consume(
        queue=queue_name,
        on_message_callback=callback
    )

    channel.start_consuming()
```

# 14) Routing Key Design Strategy

Keep routing keys semantic.

---

## Good Pattern

```text id="rkgood"
domain.action
domain.entity.action
```

Examples:

```text id="rkgood2"
payment.success
campaign.completed
contact.import.started
workflow.node.executed
```

---

---

## Why?

Easy filtering.

Flexible topic bindings.

# 15) Should Queue Be in Producer Payload?

Not usually.

Producer should care about:

* exchange
* routing key
* payload

Queue is broker-side concern.

---

---

# Why?

Because exchange decides queue.

This keeps architecture decoupled.

---

---

# 16) Production Rulebook

---

## Use Exchanges by Purpose

* topic → events
* direct → jobs
* fanout → broadcasts

---

---

## Centralize Config

Never hardcode everywhere.

---

---

## Domain-Based Routing Keys

Make events understandable.

---

---

## Separate Consumers Per Domain

Avoid mixed workloads.

---

---

## Keep Business Code Broker-Agnostic

Publish business events, not queue names.
