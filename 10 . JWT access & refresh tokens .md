# JWT Authentication in DRF — Clean Q&A Guide

---

# 1) What is an access token?

An **access token** is a short-lived credential that proves the user is authenticated.

It is sent with API requests to access protected resources.

Example:

```http id="3ab7zx"
Authorization: Bearer <access_token>
```

It usually contains:

* user identity
* permissions
* expiry time (`exp`)
* token type

Its purpose is to allow secure access without storing sessions on the server.

---

# 2) What is a refresh token?

A **refresh token** is a long-lived credential used only to obtain a new access token.

It is **not** used to access protected APIs directly.

Instead, it is sent to a refresh endpoint.

Example:

```http id="8nm4pq"
POST /api/token/refresh/
{
    "refresh": "<refresh_token>"
}
```

If valid, server returns a new access token.

---

# 3) Why do we need both access and refresh tokens?

Because security and usability must balance.

If access tokens lasted too long:

* stolen token = long-term account access

If tokens expired too quickly without refresh:

* users would log in constantly

So:

* **access token = short-lived**
* **refresh token = long-lived**

This limits damage while keeping users signed in.

---

# 4) When do they expire?

Typical values:

---

## Access Token

* 5 minutes
* 15 minutes
* 30 minutes
* 1 hour

---

## Refresh Token

* 7 days
* 30 days
* 90 days

---

Configured in Django:

```python id="e5tr9m"
SIMPLE_JWT = {
    "ACCESS_TOKEN_LIFETIME": timedelta(minutes=15),
    "REFRESH_TOKEN_LIFETIME": timedelta(days=7),
}
```

---

# 5) What happens when an access token expires?

When sent in an API request:

* server checks token
* sees `exp` has passed
* rejects request

Response:

```http id="h7zx1q"
401 Unauthorized
```

Client should then use refresh token.

---

# 6) What happens when a refresh token expires?

Then it can no longer generate new access tokens.

User must log in again.

So:

* expired access token → refresh possible
* expired refresh token → login required

---

# 7) What is `REST_FRAMEWORK` in Django?

`REST_FRAMEWORK` is a **settings dictionary**.

It is not a class.

It tells DRF which authentication and permission systems to use.

Example:

```python id="x9pw2l"
REST_FRAMEWORK = {
    "DEFAULT_AUTHENTICATION_CLASSES": (
        "rest_framework_simplejwt.authentication.JWTAuthentication",
    ),
}
```

So `REST_FRAMEWORK` is configuration only.

---

# 8) Which class checks access token presence?

The main class is:

```python id="p4mn8a"
JWTAuthentication
```

From:

```python id="z7kl2r"
rest_framework_simplejwt.authentication
```

This class:

* reads `Authorization` header
* extracts token
* validates token
* loads user

---

# 9) How does `JWTAuthentication` know token is expired?

JWT contains an expiry claim:

```json id="u6vq3d"
{
  "exp": 1770000000
}
```

During validation:

* token is decoded
* `exp` is compared to current time

If current time > `exp`, token is expired.

---

# 10) What happens if access token is expired?

Inside validation:

* `TokenError` / `InvalidToken` occurs
* `JWTAuthentication` raises `AuthenticationFailed`

Result:

```http id="f2rw9b"
401 Unauthorized
```

Request is denied.

---

# 11) Does `JWTAuthentication` refresh tokens automatically?

No.

It only validates access tokens.

Refreshing is handled separately.

---

# 12) How is refresh token handled?

Through a dedicated endpoint.

Usually:

```python id="m3sx8n"
TokenRefreshView
```

Client sends:

```json id="q8dh1z"
{
  "refresh": "<refresh_token>"
}
```

Server validates it.

If valid:

* returns new access token

Sometimes also returns new refresh token if rotation enabled.

---

# 13) Is refresh token checked by `JWTAuthentication`?

No.

That class handles access tokens for normal API requests.

Refresh tokens are handled by:

```python id="v1ta6p"
TokenRefreshView
```

and related serializer classes.

Different flow entirely.

---

# 14) Who checks if refresh token is expired?

The refresh token object itself during validation.

Roughly:

```python id="n5ly2e"
RefreshToken(refresh_string)
```

This checks:

* signature
* expiry
* token type

If expired:

* raises exception

---

# 15) What happens if refresh token is expired?

Server rejects refresh request.

Response:

```http id="r3cm7q"
401 Unauthorized
```

or

```json id="j8uw4n"
{
  "detail": "Token is invalid or expired",
  "code": "token_not_valid"
}
```

User must log in again.

---

# 16) What happens if refresh token is valid?

Then:

* new access token is created
* returned to client

Example:

```json id="d7oq6v"
{
  "access": "new_access_token"
}
```

If rotation enabled:

```json id="g4bk9s"
{
  "access": "new_access_token",
  "refresh": "new_refresh_token"
}
```

---

# 17) Which classes are involved in access-token validation?

---

## For normal API request

### 1. `JWTAuthentication`

Entry point for request authentication.

---

### 2. `AccessToken`

Represents access token and validates it.

---

### 3. `TokenBackend`

Handles decoding and signature verification.

---

### 4. `AuthenticationFailed`

Raised if invalid/expired.

---

---

# 18) Which classes are involved in refresh-token flow?

---

## For token refresh endpoint

### 1. `TokenRefreshView`

Receives refresh request.

---

### 2. `TokenRefreshSerializer`

Validates request payload.

---

### 3. `RefreshToken`

Represents refresh token and checks expiry.

---

### 4. `TokenBackend`

Decodes token.

---

### 5. `InvalidToken`

Raised if refresh invalid/expired.

---

# 19) Full access-token request flow

```text id="f9u2me"
Client Request
     ↓
JWTAuthentication.authenticate()
     ↓
Read Authorization Header
     ↓
Extract Access Token
     ↓
AccessToken() validation
     ↓
Check Signature + Expiry
     ↓
Load User
     ↓
request.user assigned
     ↓
View executes
```

---

# 20) Full refresh-token flow

```text id="w6pr8x"
Client sends refresh token
     ↓
TokenRefreshView
     ↓
TokenRefreshSerializer
     ↓
RefreshToken() validation
     ↓
Check Signature + Expiry
     ↓
Generate new access token
     ↓
Return to client
```

---

# 21) What if refresh token expired?

```text id="y4cs7l"
RefreshToken() validation
     ↓
Expiry check fails
     ↓
InvalidToken raised
     ↓
401 Unauthorized
     ↓
User must log in again
```

---

# 22) Where does expiry checking actually happen?

In token validation classes.

For access token:

```python id="s2nk8w"
AccessToken(...)
```

For refresh token:

```python id="b5te1m"
RefreshToken(...)
```

Both inspect:

```python id="c8lr4q"
exp
```

claim inside JWT payload.

---

# 23) Is expiry checking inside `REST_FRAMEWORK`?

No.

`REST_FRAMEWORK` only points to authentication classes.

Actual checking happens inside token classes and views/serializers.


```text id="h1pz6a"
        ↓
```

# 25) Cleanest understanding in one line

**Access token protects API access.
Refresh token renews authentication.
Authentication classes validate access tokens.
Refresh views validate refresh tokens.
Expiry is checked inside token classes—not in settings.**
