A **production-ready custom JWT setup** in Django REST Framework usually means:

* your own authentication class
* proper error handling
* token decoding/validation
* user lookup
* optional support for token revocation / claims
* integrated globally via `REST_FRAMEWORK`

That is the cleanest architecture for APIs.

---

# Production-Ready Custom JWT Authentication

```python
import jwt
from jwt import ExpiredSignatureError, InvalidTokenError
from django.conf import settings
from django.contrib.auth import get_user_model
from rest_framework.authentication import BaseAuthentication
from rest_framework.exceptions import AuthenticationFailed


User = get_user_model()


class CustomJWTAuthentication(BaseAuthentication):
    """
    Production-ready custom JWT authentication for DRF.
    """

    keyword = "Bearer"

    def authenticate(self, request):
        auth_header = request.headers.get("Authorization")

        if not auth_header:
            return None   # no credentials provided

        parts = auth_header.split()

        if len(parts) != 2 or parts[0] != self.keyword:
            raise AuthenticationFailed("Invalid Authorization header format")

        token = parts[1]

        try:
            payload = jwt.decode(
                token,
                settings.JWT_SECRET_KEY,
                algorithms=["HS256"]
            )

            user_id = payload.get("user_id")

            if not user_id:
                raise AuthenticationFailed("Invalid token payload")

            user = User.objects.get(id=user_id)

            if not user.is_active:
                raise AuthenticationFailed("User account is inactive")

        except ExpiredSignatureError:
            raise AuthenticationFailed("Access token expired")

        except User.DoesNotExist:
            raise AuthenticationFailed("User not found")

        except InvalidTokenError:
            raise AuthenticationFailed("Invalid access token")

        return (user, token)

    def authenticate_header(self, request):
        return self.keyword
```

# Add It in `settings.py`

```python
REST_FRAMEWORK = {
    "DEFAULT_AUTHENTICATION_CLASSES": (
        "yourapp.authentication.CustomJWTAuthentication",
    ),
    "DEFAULT_PERMISSION_CLASSES": (
        "rest_framework.permissions.IsAuthenticated",
    ),
}
```

# When Does `REST_FRAMEWORK` Come Into Action?

This is the key.

It is used **only when DRF handles a request**.

That means:

* `APIView`
* `GenericAPIView`
* `ViewSet`
* `@api_view`

All these trigger DRF’s internal pipeline.

---

---

# Example Flow

```text
Incoming Request
      ↓
Django Middleware
      ↓
URL Resolver
      ↓
DRF View (APIView / ViewSet)
      ↓
Reads REST_FRAMEWORK settings
      ↓
Runs DEFAULT_AUTHENTICATION_CLASSES
      ↓
Runs DEFAULT_PERMISSION_CLASSES
      ↓
Executes View Logic
      ↓
Returns Response
```

---

So `REST_FRAMEWORK` settings are **not global Django settings for everything**.

They only affect DRF-managed endpoints.

---

# If Normal Django View?

Example:

```python
def home(request):
    ...
```

Then `REST_FRAMEWORK` settings do nothing.

Because that is plain Django.

---

# If DRF View?

Example:

```python
from rest_framework.views import APIView
```

Then `REST_FRAMEWORK` settings apply.

# When Your Custom JWT Runs

Your class runs during:

```python
APIView.initial()
```

inside DRF.

Specifically:

```text
perform_authentication()
      ↓
for each authentication class:
      authenticate(request)
```

That’s when your `authenticate()` method is called.

---

# Missing Token Behavior

If no token:

```python
return None
```

Then DRF continues.

If endpoint requires auth → permission fails.

If endpoint allows public → request proceeds.

---

# Final Production Advice

If building custom JWT:

* keep logic inside authentication class
* avoid middleware duplication
* centralize in `REST_FRAMEWORK`
* use permissions separately

That’s the scalable design.

---

# One-Line Truth

**Middleware is for request system behavior.
`REST_FRAMEWORK` authentication is for API user identity.**
