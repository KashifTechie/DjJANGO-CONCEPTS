# 📘 Object-Oriented Programming (OOP) in JavaScript

Object-Oriented Programming (OOP) in JavaScript is a way to structure code using **objects and classes**, so you can model real-world entities and reuse logic cleanly.

Let’s break it down **practically and systematically** 👇

---

# 🔷 1. Objects (Core of JS OOP)

In JavaScript, everything revolves around objects.

```javascript
const user = {
    name: "Kashif",
    age: 22,
    greet() {
        return `Hello ${this.name}`;
    }
};
```

### Key Concepts

* **Properties** → `name`, `age`
* **Methods** → `greet()`
* `this` → refers to current object

---

# 🔷 2. Classes (Blueprint)

ES6 introduced `class` syntax (syntactic sugar over prototypes).

```javascript
class User {
    constructor(name, age) {
        this.name = name;
        this.age = age;
    }

    greet() {
        return `Hello ${this.name}`;
    }
}

const u1 = new User("Kashif", 22);
```

---

## 🔸 2.1 Constructor (VERY IMPORTANT)

* Special method called automatically on object creation
* Initializes object properties

```javascript
constructor(name, age) {
    this.name = name;
    this.age = age;
}
```

✔ Runs when `new User()` is called
✔ Only one constructor per class

---

## 🔸 2.2 What `new` Actually Does

```javascript
const u1 = new User("Kashif", 22);
```

Internally:

1. Creates empty object `{}`
2. Sets prototype → `User.prototype`
3. Binds `this` to new object
4. Calls constructor
5. Returns object

---

## 🔸 2.3 Instance vs Prototype Methods

```javascript
class Example {
    constructor() {
        this.instanceMethod = function () {};
    }

    protoMethod() {}
}
```

| Type             | Stored In   | Memory      |
| ---------------- | ----------- | ----------- |
| Instance Method  | Each object | ❌ Heavy     |
| Prototype Method | Shared      | ✅ Efficient |

---

## 🔸 2.4 Static Methods

```javascript
class MathUtil {
    static add(a, b) {
        return a + b;
    }
}
```

✔ Called on class, not object
✔ `MathUtil.add()`

---

## 🔸 2.5 Getters & Setters

```javascript
class User {
    constructor(name) {
        this._name = name;
    }

    get name() {
        return this._name;
    }

    set name(value) {
        if (!value) throw Error("Invalid");
        this._name = value;
    }
}
```

✔ Controlled access
✔ Validation layer

---

## 🔸 2.6 Private Fields

```javascript
class Test {
    #secret = 123;

    getSecret() {
        return this.#secret;
    }
}
```

✔ Truly private
✔ Not accessible outside class

---

# 🔷 3. Prototypes (REAL ENGINE OF JS)

JavaScript is **prototype-based**, not classical OOP.

```javascript
function User(name) {
    this.name = name;
}

User.prototype.greet = function () {
    return `Hello ${this.name}`;
};
```

---

## 🔸 Prototype Chain

```javascript
obj → Constructor.prototype → Object.prototype → null
```

✔ JS searches properties in this chain
✔ This is how inheritance works internally

---

# 🔷 4. `this` Keyword (CRITICAL)

### Depends on call-site, NOT definition

---

## Case 1: Object method

```javascript
user.greet();
```

👉 `this = user`

---

## Case 2: Normal function

```javascript
function test() {
    console.log(this);
}
```

👉 `window` (non-strict) / `undefined` (strict)

---

## Case 3: Arrow function

```javascript
const obj = {
    name: "Kashif",
    greet: () => console.log(this.name)
};
```

👉 No own `this` (lexical)

# 🔷 5. The 4 Pillars of OOP (DETAILED)

---

# 🔶 1. Encapsulation (DEEP)

### 👉 Definition:

Bundling data + methods and **restricting direct access**

---

## 🔸 Why Encapsulation?

Without it:

```javascript
acc.balance = -100000; // invalid state
```

With encapsulation → controlled access

---

## 🔸 Implementation Levels

---

### ✅ Level 1: Convention (_)

```javascript
this._balance
```

⚠️ Not truly private

---

### ✅ Level 2: Closures

```javascript
function BankAccount() {
    let balance = 0;

    this.deposit = function(amount) {
        balance += amount;
    };

    this.getBalance = function() {
        return balance;
    };
}
```

✔ Data hidden via scope

---

### ✅ Level 3: Private Fields (#) (BEST)

```javascript
class BankAccount {
    #balance = 0;

    deposit(amount) {
        if (amount <= 0) throw Error("Invalid");
        this.#balance += amount;
    }

    withdraw(amount) {
        if (amount > this.#balance) throw Error("Insufficient");
        this.#balance -= amount;
    }

    getBalance() {
        return this.#balance;
    }
}
```

---

## 🔸 Key Insight

Encapsulation is not just hiding data:

👉 It enforces **valid state + business rules**

---

# 🔶 2. Abstraction (DEEP)

### 👉 Definition:

Hide complexity, expose only essential interface

---

## 🔸 Example

```javascript
class Payment {
    process(amount) {
        this.#validate(amount);
        this.#charge(amount);
        this.#notify();
    }

    #validate(amount) {}
    #charge(amount) {}
    #notify() {}
}
```

---

## 🔸 Real Meaning

User only knows:

```javascript
payment.process(1000);
```

NOT:

* how validation works
* how API calls happen
* how retries happen

---

## 🔸 Abstraction vs Encapsulation

| Concept       | Focus             |
| ------------- | ----------------- |
| Encapsulation | Data protection   |
| Abstraction   | Hiding complexity |

---

# 🔶 3. Inheritance (DEEP)

### 👉 Definition:

One class derives from another

```javascript
class Animal {
    constructor(name) {
        this.name = name;
    }

    speak() {
        console.log("Animal sound");
    }
}

class Dog extends Animal {
    speak() {
        console.log("Dog barks");
    }
}
```

---

## 🔸 `super` Keyword

```javascript
constructor(name) {
    super(name);
}
```

✔ Calls parent constructor
✔ Required before using `this`

---

## 🔸 Types of Inheritance (JS Reality)

JS officially supports:

### 1. Single Inheritance

```javascript
Dog → Animal
```

---

### 2. Multilevel Inheritance

```javascript
class A {}
class B extends A {}
class C extends B {}
```

---

### 3. Hierarchical Inheritance

```javascript
class Animal {}
class Dog extends Animal {}
class Cat extends Animal {}
```

---

### ❌ NOT supported directly

* Multiple inheritance (like Java/C++)
* Achieved via **mixins**

---

## 🔸 Mixins (Simulating Multiple Inheritance)

```javascript
const canFly = {
    fly() {
        console.log("Flying");
    }
};

Object.assign(Dog.prototype, canFly);
```

---

## 🔸 Key Insight

Inheritance = **code reuse + hierarchy**

BUT overusing it → tight coupling (bad design)

---

# 🔶 4. Polymorphism (DEEP)

### 👉 Definition:

Same interface → different behavior

---

## 🔸 Type 1: Method Overriding

```javascript
class Animal {
    speak() {
        console.log("Animal sound");
    }
}

class Dog extends Animal {
    speak() {
        console.log("Bark");
    }
}
```

---

## 🔸 Type 2: Runtime Polymorphism

```javascript
const animals = [new Dog(), new Animal()];

animals.forEach(a => a.speak());
```

✔ Same method call
✔ Different output

---

## 🔸 Type 3: Duck Typing (VERY IMPORTANT in JS)

```javascript
class Car {
    move() {
        console.log("Driving");
    }
}

class Bird {
    move() {
        console.log("Flying");
    }
}

function start(entity) {
    entity.move();
}
```

✔ No inheritance needed
✔ Behavior matters, not type

---

## 🔸 Type 4: Method Overloading (NOT REAL in JS)

```javascript
function add(a, b) {
    if (b === undefined) return a;
    return a + b;
}
```

✔ Simulated manually

---

## 🔸 Key Insight

JavaScript polymorphism is mostly:

👉 **dynamic + flexible (duck typing)**
