# JWT in Django — From Beginner to Advanced

I'll explain this as if you're building an authentication system yourself.

---

# 1. Why Do We Need Authentication?

Suppose a user logs in:

```text
username = john
password = secret123
```

Server verifies credentials.

Now the question is:

> How does the server know that future requests are coming from John?

Two common approaches:

1. Session Authentication
2. JWT Authentication

---

# 2. Session Authentication (Traditional)

## Login

User sends:

```http
POST /login
```

Server verifies credentials.

Server creates:

```python
session_id = "abc123xyz"
```

and stores in DB/cache:

```text
session_id      user
----------------------
abc123xyz       john
```

Then server sends:

```http
Set-Cookie: sessionid=abc123xyz
```

Browser stores cookie.

---

## Next Request

Browser automatically sends:

```http
Cookie: sessionid=abc123xyz
```

Server:

```python
session = Session.objects.get(
    session_id="abc123xyz"
)
```

Finds John.

Request is authenticated.

---

# Session Authentication Flow

```text
Login
  |
  V
Server creates Session
  |
  V
Store in DB/Redis
  |
  V
Send Session ID to Browser
  |
  V
Browser sends Session ID
  |
  V
Server checks DB every request
```

---

# Problems with Sessions

Imagine:

```text
1 Million users
```

Server stores:

```text
1 Million session rows
```

Need:

* DB lookup every request
* Shared session storage for multiple servers
* Session cleanup

This becomes expensive.

---

# JWT Solves This

Instead of storing session data:

```text
Store user information inside token itself
```

JWT is:

```text
JSON Web Token
```

Example:

```text
eyJhbGciOiJIUzI1Ni...
```

It contains enough information to identify the user.

Server doesn't need DB lookup.

This is called:

```text
Stateless Authentication
```

---

# 3. What Exactly is JWT?

JWT has 3 parts:

```text
HEADER.PAYLOAD.SIGNATURE
```

Example:

```text
xxxxx.yyyyy.zzzzz
```

---

# Part 1: Header

Example:

```json
{
  "alg": "HS256",
  "typ": "JWT"
}
```

Meaning:

```text
alg = signing algorithm
typ = JWT
```

Encoded:

```text
eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9
```

---

# Part 2: Payload

Contains user information.

Example:

```json
{
  "user_id": 5,
  "username": "john",
  "exp": 1750000000
}
```

Encoded:

```text
eyJ1c2VyX2lkIjo1...
```

---

# Part 3: Signature

Created using:

```text
header + payload + secret key
```

Example:

```python
signature = HMAC(
    secret_key,
    header.payload
)
```

Result:

```text
abcxyz123...
```

---

Final JWT:

```text
header.payload.signature
```

Example:

```text
eyJhb...
.
eyJ1c...
.
abcxyz...
```

---

# 4. How Django Creates JWT

Using SimpleJWT:

```python
from rest_framework_simplejwt.tokens import RefreshToken

refresh = RefreshToken.for_user(user)

access = str(refresh.access_token)
refresh = str(refresh)
```

You get:

```json
{
  "access": "....",
  "refresh": "...."
}
```

---

# 5. What is Inside Access Token?

Example payload:

```json
{
  "token_type": "access",
  "user_id": 5,
  "exp": 1718000000,
  "iat": 1717990000
}
```

Fields:

| Field      | Meaning     |
| ---------- | ----------- |
| token_type | access      |
| user_id    | user id     |
| exp        | expiry time |
| iat        | issued at   |

---

# 6. What is Inside Refresh Token?

Example:

```json
{
  "token_type": "refresh",
  "user_id": 5,
  "exp": 1718600000
}
```

Usually:

```text
Access = 5 minutes
Refresh = 7 days
```

---

# 7. If JWT Is Not Stored in DB, How Is It Verified?

This is the most important concept.

Suppose token arrives:

```text
header.payload.signature
```

Server extracts:

```text
header
payload
signature
```

Then server recreates signature:

```python
new_signature = HMAC(
    SECRET_KEY,
    header.payload
)
```

Compare:

```python
if new_signature == signature:
    valid
else:
    invalid
```

---

# Why Can't Hackers Modify It?

Suppose attacker changes:

```json
{
  "user_id": 5
}
```

to

```json
{
  "user_id": 1
}
```

Payload changes.

Now signature becomes invalid.

Server calculates:

```python
expected_signature != received_signature
```

Request rejected.

---

# 8. What Happens on Every Request?

User sends:

```http
Authorization: Bearer ACCESS_TOKEN
```

Django JWT authentication class:

```python
JWTAuthentication
```

does:

### Step 1

Extract token.

### Step 2

Verify signature.

### Step 3

Check expiry.

### Step 4

Extract user_id.

### Step 5

Load user.

```python
user = User.objects.get(id=user_id)
```

### Step 6

Set:

```python
request.user
```

Authenticated.

---

# 9. How Does It Know Access Token Expired?

JWT payload contains:

```json
{
  "exp": 1718000000
}
```

Current time:

```python
now = 1718005000
```

Check:

```python
if now > exp:
    raise TokenExpired
```

Authentication fails.

Response:

```http
401 Unauthorized
```

---

# 10. What Happens When Access Token Expires?

Flow:

```text
Access Token Expired
        |
        V
Frontend receives 401
        |
        V
Send Refresh Token
        |
        V
Request New Access Token
```

Example:

```http
POST /token/refresh/
```

Body:

```json
{
  "refresh": "xxxxx"
}
```

---

# 11. Django SimpleJWT Refresh Process

Endpoint:

```python
/token/refresh/
```

Receives:

```json
{
  "refresh": "..."
}
```

Checks:

### Signature

Valid?

### Expiry

Expired?

### Token Type

Refresh token?

If all pass:

Create new access token.

Return:

```json
{
  "access": "new_token"
}
```

---

# 12. How Is New Access Token Generated?

SimpleJWT reads:

```json
{
  "user_id": 5
}
```

from refresh token.

Creates:

```json
{
  "token_type": "access",
  "user_id": 5,
  "exp": new_time
}
```

Signs it.

Returns new access token.

---

# 13. What Happens If Refresh Token Expired?

Suppose:

```text
Refresh lifetime = 7 days
```

Day 8:

```http
POST /token/refresh/
```

SimpleJWT checks:

```python
now > exp
```

Result:

```text
Refresh token expired
```

Response:

```http
401 Unauthorized
```

Example:

```json
{
  "detail": "Token is expired"
}
```

---

# 14. Then What?

User must login again.

Flow:

```text
Refresh Expired
      |
      V
Cannot Generate Access Token
      |
      V
Force Login Screen
      |
      V
User Logs In Again
      |
      V
New Access Token
New Refresh Token
```

---

# Complete JWT Lifecycle

```text
User Login
    |
    V
Create Access Token (5 min)
Create Refresh Token (7 days)
    |
    V
Client Stores Tokens
    |
    V
Request API using Access Token
    |
    V
Access Valid ?
    |
   Yes ----------------------> Continue
    |
   No
    |
    V
Use Refresh Token
    |
    V
Refresh Valid ?
    |
   Yes
    |
    V
Generate New Access Token
    |
    V
Continue
    |
   No
    |
    V
Refresh Expired
    |
    V
Login Again
```

# Session vs JWT Summary

| Feature                 | Session     | JWT             |
| ----------------------- | ----------- | --------------- |
| Stored in DB            | Yes         | No              |
| Stateful                | Yes         | No              |
| DB lookup every request | Yes         | Usually No      |
| Easy logout             | Yes         | Harder          |
| Scales better           | No          | Yes             |
| Mobile APIs             | Less common | Very common     |
| DRF APIs                | Less common | Standard choice |

For Django REST Framework APIs, the most commonly used package today is [Simple JWT Documentation](https://django-rest-framework-simplejwt.readthedocs.io/?utm_source=chatgpt.com), which implements the access-token/refresh-token flow described above.

A next step would be understanding **JWT blacklisting**, because that answers a very important question:

> "If JWTs are not stored in the database, how does logout work?"

That's usually the first advanced concept people encounter after learning the basics.
