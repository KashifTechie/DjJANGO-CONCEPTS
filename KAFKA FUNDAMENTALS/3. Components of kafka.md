## What is a Topic?

* A **Topic** is a logical name for a stream of data.
* Producers write data into a topic.
* Consumers read data from a topic.
* Similar to a database table — but append-only.

### Example

```
Topic: orders
Topic: payments
Topic: user-events
```

Kafka does NOT store data directly at topic level.
Topic is divided into **Partitions** (real storage units).

---

# 2️⃣ Partition (Physical Storage Unit)

## What is a Partition?

* A Topic is split into multiple partitions.
* Each partition is:

  * An ordered
  * Immutable
  * Append-only log

---

## Example Topic with 3 Partitions

```
Topic: orders

Partition 0
[0] Order-A
[1] Order-B

Partition 1
[0] Order-X
[1] Order-Y

Partition 2
[0] Order-M
[1] Order-N
```

### Important Rules

* Order is guaranteed **inside one partition**
* Order is NOT guaranteed across partitions
* Each partition lives on a broker

---

## Why Partitions Exist?

1. Scalability
2. Parallel processing
3. Distributed storage

More partitions → more throughput

---

# 3️⃣ Offset (Message Position)

## What is Offset?

* Offset = Unique number of message inside a partition
* Assigned automatically by Kafka
* Starts from 0

---

## Example

```
Partition 0

Offset 0 → Order-A
Offset 1 → Order-B
Offset 2 → Order-C
```

Consumers track offsets like this:

```
Last committed offset = 1
Next read = offset 2
```

### Important

* Kafka does NOT delete message after read
* Offset allows replay
* Offset is per partition

---

# 4️⃣ Broker (Kafka Server)

## What is a Broker?

* A broker = one Kafka server
* Stores partitions
* Handles read/write requests
* Has unique broker.id

---

## Example with 3 Brokers

```
Broker 1
Broker 2
Broker 3
```

Partitions are distributed across brokers.

---

# Topic + Partition + Replicas Across Brokers

Now let’s combine everything.

---

## Topic: orders

Partitions: 3
Replication factor: 2

```
Broker 1          Broker 2          Broker 3
------------------------------------------------
P0 (Leader)       P1 (Leader)       P2 (Leader)
P1 (Follower)     P2 (Follower)     P0 (Follower)
```

### What This Means

Partition 0:

* Leader → Broker 1
* Follower → Broker 3

Partition 1:

* Leader → Broker 2
* Follower → Broker 1

Partition 2:

* Leader → Broker 3
* Follower → Broker 2

---

# 🔥 How Write Happens

Producer sends message to:

```
Producer
    |
    v
Leader of that partition
```

Example:

```
Producer → Partition 0 Leader → Broker 1
```

Then:

```
Broker 1 (Leader)
    |
    v
Broker 3 (Follower replicates)
```

Only after replication (based on acks setting), producer gets acknowledgment.

---

# 💥 What Happens If One Broker Goes Down?

### Suppose Broker 1 crashes

```
Broker 1 ❌
```

Partitions affected:

* P0 leader was on Broker 1
* P1 follower was on Broker 1

---

## Step-by-Step Recovery

1. Controller detects Broker 1 failure
2. For Partition 0:

   * Broker 3 was follower
   * Broker 3 becomes new Leader
3. Producers automatically switch to new leader
4. Consumers continue reading

New layout:

```
Broker 2          Broker 3
--------------------------------
P1 (Leader)       P0 (Leader)
P2 (Follower)     P2 (Leader)
```

System continues working.

This is why replication exists.

# 5️⃣ Cluster (Full System)

## What is a Cluster?

* A group of brokers working together
* Looks like one system to clients
* Handles:

  * Metadata
  * Leader election
  * Partition distribution

---

## Cluster View

```
                Kafka Cluster
------------------------------------------------
| Broker1 | Broker2 | Broker3 | Broker4 |
------------------------------------------------
```

Clients connect to cluster (not specific broker permanently).

Cluster responsibilities:

* Manage topic metadata
* Track partition leaders
* Detect broker failures
* Reassign partitions

---

# 6️⃣ Producer (After Understanding Core Storage)

## What Producer Does

1. Connects to cluster
2. Fetches metadata
3. Decides partition
4. Sends to leader
5. Waits for acknowledgment

Partition selection methods:

```
hash(key) % number_of_partitions
```

OR

```
Round robin
```

---

# 7️⃣ Consumer

## What Consumer Does

* Subscribes to topic
* Fetches metadata
* Reads from partition leader
* Tracks offset
* Commits offset

Kafka uses **pull model**:

```
Consumer → Broker (give me data)
```

---

# 8️⃣ Consumer Group (Parallel Processing)

Example:

Topic has 4 partitions

```
P0
P1
P2
P3
```

Consumer Group has 2 consumers

```
Consumer1 → P0, P1
Consumer2 → P2, P3
```

If Consumer1 crashes:

```
Consumer2 → P0, P1, P2, P3
```

Automatic rebalance happens.

---

# 9️⃣ Log Segments (Disk Storage)

Partition stored as files:

```
orders-0/
  000000.log
  000001.log
```

When size limit reached:

```
Segment1
Segment2
Segment3
```

Old segments deleted based on retention.

---

# 🔟 Retention Policy

Two types:

Time-based:

```
retention.ms = 7 days
```

Size-based:

```
retention.bytes = 10GB
```

After limit → old segments deleted.

---

# 1️⃣1️⃣ Log Compaction

Used when key matters.

Example:

```
user1 → balance 100
user1 → balance 200
```

After compaction:

```
user1 → balance 200
```

Keeps only latest per key.

---

# 1️⃣2️⃣ Controller (Cluster Brain)

In modern Kafka (KRaft mode):

Responsible for:

* Leader election
* Broker monitoring
* Partition reassignment

If broker fails:

```
Detect → Elect new leader → Update metadata
```
