# 🚀 What is Apache Kafka?

Apache Kafka is a **distributed event streaming platform** used to send, store, and process large amounts of messages (events) in real-time.

It works using 3 main components:

* **Producer** → Sends data
* **Broker** → Stores & manages data
* **Consumer** → Reads & processes data

Now let's go step-by-step in deep technical detail.

---

# 1️⃣ What Exactly Happens When a Producer Sends Data?

---

## 🟢 Step-by-Step: Producer Flow (Line by Line)

### 1. Producer starts

* Loads Kafka config:

  * `bootstrap.servers`
  * `acks`
  * `retries`
  * `linger.ms`
  * `batch.size`
  * `key.serializer`
  * `value.serializer`

---

### 2. Producer connects to Kafka cluster

* Connects to one broker from `bootstrap.servers`
* Requests **metadata**

  * Which brokers exist?
  * Which partitions exist?
  * Who is leader for each partition?

Broker responds with cluster metadata.

---

### 3. Producer prepares the message

When you call:

```python
producer.send("orders", key="user1", value="order123")
```

Internally:

1. Serialize key & value
2. Decide partition:

   * If key present → hash(key) % partitions
   * If no key → round robin

---

### 4. Message goes into Producer Buffer

Producer does NOT immediately send message.

It puts it into an in-memory buffer.

Messages are sent when:

* batch size full OR
* linger.ms time reached OR
* buffer full OR
* flush() called

This improves throughput.

---

### 5. Producer sends batch to Broker

Producer sends a **ProduceRequest** to the leader broker of that partition.

Request contains:

* Topic
* Partition
* Batch of messages
* Required ACK level

---

## 🔍 Deep Dive: Batch Send Triggers

### 1️⃣ batch.size full

`batch.size` = maximum size (in bytes) of a batch per partition.

Producer groups multiple messages together into one batch.

If total messages in that batch reach this size → producer sends it to broker.

**Example:**

```
batch.size = 16KB
If accumulated messages reach 16KB → send immediately.
```

### 2️⃣ linger.ms time reached

`linger.ms` = how long producer waits before sending a batch.

Even if batch is NOT full, producer waits this many milliseconds.

If time expires → batch is sent.

**Example:**

```
linger.ms = 10ms
Producer waits 10ms to collect more messages.
After 10ms → sends whatever is available.
```

This improves batching efficiency.

### 3️⃣ buffer full

Producer has a total memory buffer (`buffer.memory`).

If producer keeps sending messages but broker is slow, memory fills up.

When buffer is full:

> Producer blocks (or throws exception depending on config).

This is **backpressure**.

### 4️⃣ flush() called

If application explicitly calls:

```python
producer.flush()
```

Producer immediately sends all pending batches to broker.

Used when:

> You want to guarantee all messages are sent before shutdown.

---

## ✅ Who Stores Data on Disk?

> 👉 Producer NEVER stores data on disk.  
> 👉 Broker stores data on disk.

**Step-by-step:**

1. Producer receives message
2. Message stored in **Producer memory buffer**
3. Batch sent to Broker
4. Broker stores message in its **memory (page cache)**
5. Broker writes message to its **local disk (log segment file)**

| Component | Storage |
|-----------|---------|
| Producer  | Memory only |
| Broker    | Memory + Disk |

---

## ✅ When Data Is Written to Disk — Who Removes It From Memory?

### Producer Side

Once:

* Broker sends acknowledgment (ACK)
* Producer marks batch as complete
* Producer removes it from memory buffer

> Producer does NOT store data on disk.

### Broker Side

Broker:

* Appends message to its in-memory page cache
* OS writes it to disk (log file)
* Broker does NOT manually remove it from memory

> Memory management is handled by the **Operating System page cache**.  
> Kafka relies on OS for disk flushing and memory eviction.

---

## 🔥 Summary: Producer Flow (Rewritten Clearly)

### 4️⃣ Message Goes Into Producer Buffer

Producer does NOT immediately send the message.

It stores the message in its **in-memory buffer**.

Messages are grouped into batches per partition.

A batch is sent when:

* `batch.size` is reached (batch becomes full)
* `linger.ms` timeout expires
* Producer memory buffer becomes full
* `flush()` is called manually

This batching improves throughput and reduces network calls.

> Producer only keeps messages in memory, never on disk.  
> After broker acknowledges the batch, producer removes it from memory.

### 5️⃣ Producer Sends Batch to Broker

Producer sends a **ProduceRequest** to the leader broker of that partition.

The request contains:

* Topic
* Partition
* Batch of messages
* Required acknowledgment level (`acks`)

Broker receives the batch → Broker appends messages to its **in-memory page cache** → Broker writes the messages to its **local disk (log segment file)** → After writing (depending on ACK configuration), broker sends acknowledgment to producer → Producer then clears that batch from its memory buffer.

# 2️⃣ What Broker Does When Message Arrives

---

## 🟡 Case A: Single Message Comes

### Step 1: Broker receives request

Broker checks:

* Is this broker leader for partition?

  * If NO → redirect error
  * If YES → continue

---

### Step 2: Append to Log

Kafka stores data in:

```
topic → partition → log file
```

Broker:

* Appends message at end of log file
* Assigns offset number
* Writes to OS page cache
* Optionally flushes to disk (based on config)

---

### Step 3: Replication (if replicas exist)

If replication factor > 1:

* Leader sends message to follower brokers
* Followers append to their logs
* Followers send ACK to leader

---

### Step 4: Broker Sends Acknowledgment to Producer

Depends on `acks` setting:

#### 🔹 acks=0

* No response
* Producer doesn't know success or failure

#### 🔹 acks=1

* Leader writes message
* Immediately ACKs producer
* Followers may not have replicated yet

#### 🔹 acks=all (strongest guarantee)

* Leader waits until all in-sync replicas confirm
* Then sends success ACK to producer

---

## 🟡 Case B: Many Messages (Batch Mode)

When batch of 1000 messages arrives:

Broker:

* Appends entire batch sequentially
* Assigns consecutive offsets
* Replicates batch to followers
* Sends single ACK for entire batch

> Kafka is optimized for sequential disk writes → very fast.

---

## 🟡 Case C: Message is Waiting (High Load Scenario)

If broker is overloaded:

* Request enters network queue
* Then request queue
* Then IO thread processes it

Kafka uses:

* Thread pool for network
* Thread pool for disk IO

If queue fills:

* Broker may throttle producer
* Or return timeout
* Producer retries automatically

---

# 3️⃣ What Happens Inside Consumer

---

## 🔵 Step-by-Step: Consumer Flow

### 1. Consumer starts

Config includes:

* `group.id`
* `auto.offset.reset`
* `enable.auto.commit`
* `max.poll.records`

---

### 2. Consumer joins a Consumer Group

Broker assigns:

* Partitions to each consumer in group
* Uses group coordinator

> Only ONE consumer reads one partition.

---

### 3. Consumer sends Fetch Request

Consumer sends:

```
FetchRequest(topic, partition, offset)
```

Broker:

* Reads from log starting at offset
* Returns batch of messages

---

### 4. Consumer Processes Messages

Consumer:

* Deserializes
* Executes business logic

  * Save to DB
  * Call API
  * Process event

---

# 4️⃣ How Consumer Notifies Broker About Success

Important concept: **Offset Commit**

> Kafka does NOT care if your business logic succeeds.

Kafka only tracks:

> What offset has consumer committed?

---

## Two Ways Offset Commit Works

### 🔹 1. Auto Commit (enable.auto.commit=true)

Consumer automatically commits latest offset every few seconds.

**Problem:**

* If processing fails after commit → data lost

---

### 🔹 2. Manual Commit (Recommended)

After processing successfully:

```python
consumer.commit()
```

This tells broker:

> "I have successfully processed up to offset X"

Broker stores offset in internal topic:

```
__consumer_offsets
```

---

# 5️⃣ What Happens If Consumer Crashes?

If offset was committed:

* New consumer resumes from next offset

If offset NOT committed:

* Message is re-delivered

This gives:

* **At least once delivery guarantee**

---

# 6️⃣ Full End-to-End Flow Summary

---

## 🟢 Producer Side

1. Connect
2. Fetch metadata
3. Serialize message
4. Decide partition
5. Buffer message
6. Batch
7. Send to leader
8. Wait for ACK

---

## 🟡 Broker Side

1. Receive request
2. Check leadership
3. Append to log
4. Assign offset
5. Replicate
6. Send ACK

---

## 🔵 Consumer Side

1. Join group
2. Get partition
3. Fetch from offset
4. Process
5. Commit offset

---

# 7️⃣ What If Only One Broker Exists And It Crashes?

If:

* Replication factor = 1
* Single broker cluster

Then:

* Data stored only on that broker
* If disk lost → data lost
* If broker restarts → data safe (if disk intact)

> Kafka is fault tolerant only if:
> * Replication factor ≥ 2
> * Multiple brokers exist

# 8️⃣ Important Internal Concepts

| Concept        | Meaning                      |
| -------------- | ---------------------------- |
| Offset         | Unique position in partition |
| Partition      | Ordered message log          |
| Leader         | Handles reads & writes       |
| Follower       | Replica copy                 |
| ISR            | In Sync Replicas             |
| Consumer Group | Parallel processing unit     |

---

# 9️⃣ Simple Mental Model

Think of Kafka like:

```
Producer → Append-only log → Consumer reads pointer
```

* Consumers move a pointer (offset).
* Broker never deletes data immediately.
* Messages stay for configured retention period.
