In Apache Kafka, the **message key** is used by the **producer** to decide **which partition of a topic the message should go to**.
Since each partition belongs to a **broker**, this indirectly decides **which broker will store the message**.

Let’s go step-by-step.

---

# 1️⃣ Producer Gets Metadata First

When a producer connects to Kafka:

1. Producer connects to **any broker** (bootstrap server).
2. It requests **cluster metadata**.

Metadata contains:

* topic name
* number of partitions
* which broker is **leader for each partition**

Example metadata:

| Partition | Leader Broker |
| --------- | ------------- |
| 0         | Broker-1      |
| 1         | Broker-2      |
| 2         | Broker-3      |

Now the **producer knows where each partition lives**.

---

# 2️⃣ Producer Uses the Message Key

When the producer sends a message:

```
producer.send(topic, key, value)
```

Kafka applies a **hash function to the key**.

```
partition = hash(key) % number_of_partitions
```

Example:

```
key = "user123"
hash("user123") = 4587362
partitions = 3

4587362 % 3 = 2
```

So the message goes to **partition 2**.

---

# 3️⃣ Partition Determines the Broker

From metadata:

| Partition | Broker   |
| --------- | -------- |
| 0         | Broker-1 |
| 1         | Broker-2 |
| 2         | Broker-3 |

Since partition **2** is on **Broker-3**, the producer sends the message **directly to Broker-3**.

```
Producer
   ↓
hash(key)
   ↓
partition 2
   ↓
Broker-3
```

---

# 4️⃣ Why Kafka Uses the Key

Using the same key ensures **ordering**.

Example:

Messages with key `"user123"`:

```
msg1
msg2
msg3
```

All go to **the same partition**, so Kafka guarantees **order**.

```
Partition 2
msg1
msg2
msg3
```

---

# 5️⃣ If No Key Is Provided

If the key is **null**:

Kafka uses **round-robin or sticky partitioning**.

Messages get distributed across partitions like:

```
msg1 → partition 0
msg2 → partition 1
msg3 → partition 2
msg4 → partition 0
```

This improves **load balancing**.

---

# 6️⃣ Important Point

The producer **does NOT choose the broker directly**.

Instead:

```
key
  ↓
hash(key)
  ↓
partition
  ↓
leader broker of that partition
```

---

✅ **One-line summary**

> Kafka hashes the message key to compute a partition number, and the producer sends the message to the broker that is the leader of that partition.
