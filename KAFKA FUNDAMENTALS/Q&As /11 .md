# Rebalancing in Apache Kafka

Rebalancing is the process where Kafka redistributes partitions among consumers inside the same consumer group.

---

# Why Rebalancing Exists

Kafka rule:

```text id="jlwm1s"
Inside one consumer group:
One partition → one active consumer
```

So Kafka must decide:

```text id="jlwm4d"
Which consumer gets which partition?
```

Whenever group membership changes.

---

# Example

Topic:

```text id="jlwm7s"
campaign_tasks
```

Partitions:

```text id="jlwm9g"
P0 P1 P2 P3
```

Group:

```text id="jlwm2r"
campaign_group
```

---

# Initial State

Only one consumer:

```text id="jlwm5z"
Consumer A
```

Kafka assigns:

```text id="jlwm8l"
A -> P0,P1,P2,P3
```

---

# New Consumer Joins

Consumer B starts.

Kafka detects:

```text id="jlwm0s"
Group membership changed
```

Triggers:

# REBALANCE

---

# After Rebalance

Kafka redistributes partitions:

```text id="jlwm3j"
A -> P0,P1
B -> P2,P3
```

Now load is shared.

---

# Main Situations That Trigger Rebalancing

---

# 1. New Consumer Joins

Example:

```bash id="jlwm6r"
python manage.py run_consumer --group campaign
```

again.

---

# 2. Consumer Crashes

Example:

```text id="jlwm9l"
Consumer B dies
```

Kafka reassigns B's partitions.

Before:

```text id="jlwm1y"
A -> P0,P1
B -> P2,P3
```

After:

```text id="jlwm4t"
A -> P0,P1,P2,P3
```

---

# 3. Consumer Stops Gracefully

Examples:

* deployment restart
* Ctrl+C
* pod shutdown

---

# 4. Partition Count Changes

Example:

```text id="jlwm7l"
4 partitions → 8 partitions
```

Kafka redistributes new partitions.

---

# What Happens During Rebalance

---

# Step 1

Kafka detects group change.

---

# Step 2

Consumers temporarily stop consuming.

---

# Step 3

Kafka decides new partition assignments.

---

# Step 4

Consumers resume with new partitions.

---

# Important Consequence

During rebalance:

```text id="jlwm0l"
consumption pauses briefly
```

Too many rebalances:

* reduce throughput
* increase lag
* slow system

---

# Example Flow

---

# Before

```text id="jlwm3s"
A -> P0,P1
B -> P2,P3
```

---

# Consumer C Joins

Kafka rebalances.

---

# After

```text id="jlwm6m"
A -> P0
B -> P1
C -> P2,P3
```

---

# Why Rebalancing Is Important

Without rebalancing:

* crashed consumers would lose partitions permanently
* scaling consumers would require manual assignment
* failover would not work

Rebalancing gives Kafka:

* scalability
* fault tolerance
* automatic load balancing

---

# Important Production Concepts

---

# More Consumers ≠ Always Better

Too many consumers cause:

* frequent rebalances
* coordination overhead
* slower system

---

# Stable Consumers Are Better

GOOD:

```text id="jlwm9y"
long-running stable consumers
```

BAD:

```text id="jlwm2k"
consumers constantly starting/stopping
```

because it causes rebalance storms.

---

# Best Practice

Usually:

```text id="jlwm5j"
few Kafka consumers
+
internal thread pool
```

instead of:

* huge number of Kafka consumers

---

# Your Architecture Example

Suppose:

```text id="jlwm8n"
campaign_group
```

has:

```text id="jlwm1v"
2 consumers
```

and topic has:

```text id="jlwm4b"
4 partitions
```

Kafka may assign:

```text id="jlwm7q"
Consumer 1 -> P0,P1
Consumer 2 -> P2,P3
```

If Consumer 2 crashes:

Kafka rebalances:

```text id="jlwm0w"
Consumer 1 -> P0,P1,P2,P3
```

Automatic failover.

---

# Key Mental Model

Think:

```text id="jlwm3u"
Partitions = work chunks

Consumers = workers

Rebalance = manager redistributing work
```

When:

* worker joins
* worker leaves

manager redistributes tasks automatically.
