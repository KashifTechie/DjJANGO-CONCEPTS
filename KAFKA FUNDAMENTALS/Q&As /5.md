When a **producer gets metadata from brokers**, it uses that metadata to **decide exactly where to send each message in the Kafka cluster**.

Below is the precise explanation.

---

# What Producer Does With Broker Metadata

## 1. What Metadata Contains

When a **Kafka producer connects to the cluster**, it asks a broker for **cluster metadata**.

The metadata contains:

* List of **all brokers**
* List of **topics**
* Number of **partitions per topic**
* Which **broker is leader for each partition**
* Which brokers are **replicas**

Example metadata:

```
Topic: orders

Partition 0 → Leader: Broker 2
Partition 1 → Leader: Broker 1
Partition 2 → Leader: Broker 3
```

---

# Step-by-Step What Producer Does With Metadata

## Step 1 — Producer Gets Metadata

Producer sends a **Metadata Request** to any broker.

```
Producer
   |
   | Metadata Request
   v
Broker
```

Broker replies with cluster metadata.

```
Producer receives:

Topic: orders
Partitions: 3

Partition 0 → Broker 2
Partition 1 → Broker 1
Partition 2 → Broker 3
```

Producer stores this **in memory cache**.

---

# Step 2 — Producer Chooses Partition

When you send a message:

```python
producer.send("orders", value="order123")
```

Producer must decide **which partition to send the message to**.

Kafka uses:

### Case 1 — Key Provided

If message has a **key**, Kafka hashes the key.

```
partition = hash(key) % total_partitions
```

Example:

```
key = user_42
hash(user_42) % 3 = partition 1
```

---

### Case 2 — No Key

Kafka uses **round robin** or **sticky partitioning**.

Example:

```
message1 → partition 0
message2 → partition 1
message3 → partition 2
```

---

# Step 3 — Producer Finds Leader Broker

Using metadata:

```
Partition 1 → Leader Broker 1
```

Producer now knows:

```
Send message to Broker 1
```

Important:

👉 **Producer always sends messages only to the partition leader**

Followers are **not contacted by producer**.

Replication happens **inside Kafka brokers**.

---

# Step 4 — Producer Sends Message

Producer opens a TCP connection to the correct broker.

```
Producer
   |
   | message → partition 1
   v
Broker 1 (Leader)
```

Broker writes message to the partition log.

---

# Step 5 — Broker Replicates to Followers

Broker internally replicates data.

```
Leader Broker 1
      |
      v
Follower Broker 2
Follower Broker 3
```

Producer **does not handle replication**.

---

# Step 6 — Producer Updates Metadata When Needed

Metadata is refreshed when:

* new broker added
* topic partitions changed
* leader changes
* broker fails

Example:

```
Broker 1 crashes
```

New leader elected:

```
Partition 1 → Broker 3
```

Producer refreshes metadata and updates routing.

# In One Sentence

**Producer uses metadata to know which broker is the leader of the partition so it can send messages directly to the correct broker.**

---

# Very Important Realization

Producer **does NOT send messages randomly**.

It sends messages **directly to the correct broker leader** using metadata.
