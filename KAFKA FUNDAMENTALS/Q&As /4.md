**Question**

In a Python application using Apache Kafka with the Confluent Kafka Python Client, when we call:

```python
producer.produce(...)
producer.poll(0)
```

what actually sends the data to the Kafka broker?

Some documentation says:

> `poll(0) drains the internal event queue and fires pending callbacks on your thread`

But earlier we said that **an internal sender thread sends the data**.
So which one is correct? Does `poll()` send the data, or does the background thread send it?

---

# Answer

Both statements are correct, but they refer to **two different internal mechanisms** of the Kafka producer.

The key idea is that the producer internally maintains **two different queues**:

```
1. Message Queue (for sending data)
2. Event Queue (for callbacks and events)
```

And **different components handle each queue**.

---

# 1️⃣ What Happens When `produce()` Is Called

Example:

```python
producer.produce(topic="orders", value=b"data")
```

This function **does not send data immediately**.

Instead it performs these steps:

```
1. Create a Kafka record
2. Serialize the message
3. Place the record in the producer's internal message buffer
4. Return immediately
```

So the operation is basically:

```
produce() → enqueue message in local buffer
```

No network communication happens yet.

---

# 2️⃣ What Actually Sends the Data

The **internal background I/O thread** (from the underlying C library *librdkafka*) is responsible for sending data.

This thread continuously performs the following tasks:

```
read messages from buffer
build record batches
compress batches (if enabled)
open/reuse TCP connection
send ProduceRequest to broker
wait for acknowledgements
```

So the real sender is:

```
Kafka Producer Sender Thread
```

This thread runs automatically once the producer is created.

---

# 3️⃣ What Happens After the Broker Responds

After the Kafka broker receives the batch and processes it, it sends an acknowledgement (ACK).

At that point the producer:

```
creates a delivery report event
places it in the internal event queue
```

So now there is a **pending callback event**, but it has not been executed yet.

---

# 4️⃣ What `poll(0)` Actually Does

When you call:

```python
producer.poll(0)
```

the client:

```
drains the internal event queue
executes pending callbacks
handles errors and stats events
returns immediately
```

This is exactly what the documentation means when it says:

```
poll(0) drains the internal event queue and fires pending callbacks
```

Important point:

```
poll() processes events
poll() does NOT perform network sending
```

---

# 5️⃣ Timeline Example

Example sequence of events.

```
t = 0 ms
produce()
message added to buffer
```

```
t = 10 ms
background sender thread sends batch to broker
```

```
t = 15 ms
broker sends ACK
```

```
delivery event added to event queue
```

```
t = 20 ms
producer.poll(0)
```

Now:

```
event queue drained
delivery callback executed
```

---

# 6️⃣ Why Callbacks Run in `poll()` Instead of Background Threads

The Python client intentionally runs callbacks on **your thread** when `poll()` is called.

This avoids problems like:

```
background threads modifying Python objects
race conditions
thread safety issues
```

So the design is:

```
background threads → networking
application thread → callbacks
```

---

# 7️⃣ Internal Producer Architecture

The overall flow inside the producer looks like this:

```
Application Thread
      │
      │ produce()
      ▼
Message Queue
      │
      ▼
Sender Thread (I/O thread)
      │
      │ send batch
      ▼
Kafka Broker
      │
      │ ACK
      ▼
Event Queue
      │
      ▼
poll()
      │
      ▼
Callback executed
```

---

# 8️⃣ Final Clarification

Both statements are correct but refer to different parts of the system.

```
produce() → adds message to buffer
sender thread → sends data to Kafka broker
poll() → drains event queue and executes callbacks
```

So the **actual network sending is done by the internal sender thread**, while **`poll()` only handles events and callbacks**.

---

✅ **Key takeaway**

The producer separates responsibilities:

```
Network operations → background threads
Callback execution → your application thread via poll()
```
