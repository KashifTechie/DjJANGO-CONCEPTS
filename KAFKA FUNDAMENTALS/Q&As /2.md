# ❓ Proper Question

**In Apache Kafka, when a producer sends messages to a topic that has multiple partitions:**

1. Does a **single `producer.send()` call distribute its data across multiple partitions**, or does it go to only one partition?
2. When **many producer calls are made**, how are those messages **accumulated and distributed across partitions**?
3. Once messages are stored in partitions, **how does a consumer read them**?
4. **In what order are messages consumed?**
5. **How does the consumer notify the broker that it has processed the messages?**
6. **What does the broker do after receiving that notification?**

---

# ✅ Complete Answer

---

# 1️⃣ Does One `producer.send()` Call Distribute Data Across Partitions?

No.

A **single `producer.send()` call always writes exactly one message to exactly one partition**.

Kafka **never splits a message across partitions**.

Example:

Topic:

```
orders
 ├── partition 0
 ├── partition 1
 └── partition 2
```

Producer call:

```python
producer.send("orders", value="order1")
```

Kafka will:

1. Run the **partitioner**
2. Select **one partition**
3. Write the **entire message to that partition**

Example result:

```
partition 1 → order1
```

This **never happens**:

```
partition 0 → half message
partition 1 → half message
```

Kafka messages are **atomic units**.

---

# 2️⃣ How Kafka Chooses the Partition

Kafka uses a **partitioning strategy**.

---

### Case 1 — Key Present

If a key exists:

```python
producer.send("orders", key="user1", value="order1")
```

Kafka computes:

```
partition = hash(key) % number_of_partitions
```

Example:

```
hash("user1") % 3 = 1
```

Result:

```
partition 1 → order1
```

This ensures **same key always goes to the same partition**.

---

### Case 2 — No Key

If no key is given:

```python
producer.send("orders", value="order1")
```

Kafka uses **round-robin or sticky partitioning**.

Example:

```
msg1 → partition0
msg2 → partition1
msg3 → partition2
msg4 → partition0
```

This distributes load evenly.

---

# 3️⃣ How Messages From Many Calls Get Accumulated

Messages are **not sent immediately to the broker**.

The producer first stores them in **RAM buffers called batches**.

Each partition has its **own batch buffer**.

Example producer sends 6 messages:

```
send(msg1)
send(msg2)
send(msg3)
send(msg4)
send(msg5)
send(msg6)
```

Producer partition selection:

```
msg1 → partition0
msg2 → partition1
msg3 → partition2
msg4 → partition0
msg5 → partition1
msg6 → partition2
```

Producer memory buffer becomes:

```
Producer RAM Buffer

partition0 batch → [msg1, msg4]
partition1 batch → [msg2, msg5]
partition2 batch → [msg3, msg6]
```

These batches are sent to the broker when:

```
batch.size reached
OR
linger.ms timeout reached
OR
buffer memory full
OR
producer.flush() called
```

---

# 4️⃣ How Data Is Stored Inside the Partition

On the broker disk:

```
topic/
   partition0/
       000000000000.log
       000000000000.index
       000000000000.timeindex
```

Messages are stored **sequentially**.

Example:

```
partition0 log

offset 0 → msg1
offset 1 → msg4
offset 2 → msg7
offset 3 → msg10
```

Kafka **never modifies existing messages**.

It only **appends new messages to the end**.

---

# 5️⃣ How Consumers Read Messages (Clarifying `subscribe()`)

This is where confusion usually happens.

When using **Confluent Kafka or Kafka clients**, you usually do:

```python
consumer.subscribe(["orders"])
```

Notice:

You only pass the **topic name**, not partition or offset.

So how does Kafka know what to read?

### Step 1 — Consumer joins a Consumer Group

Consumer config includes:

```
group.id = order-service
```

When the consumer subscribes:

```
consumer.subscribe(["orders"])
```

Kafka performs **consumer group coordination**.

A special broker called the **Group Coordinator** assigns partitions.

Example:

Topic:

```
orders
 ├── partition0
 ├── partition1
 └── partition2
```

Consumers:

```
consumerA
consumerB
```

Partition assignment may become:

```
consumerA → partition0
consumerA → partition1
consumerB → partition2
```

So the consumer **does not manually choose partitions**.

Kafka assigns them automatically.

---

### Step 2 — Kafka Determines the Offset to Start From

Now Kafka must decide **which offset the consumer should read from**.

Three cases exist.

---

#### Case 1 — Consumer Group Has Previous Offset

If this consumer group already consumed before:

Broker checks `__consumer_offsets`.

Example:

```
group: order-service
partition: 0
last_committed_offset = 5
```

Consumer starts reading from:

```
offset = 5
```

---

#### Case 2 — First Time Consumer Group

If no offset exists, Kafka checks:

```
auto.offset.reset
```

Config may be:

```
auto.offset.reset = earliest
```

Consumer starts from:

```
offset 0
```

If config is:

```
auto.offset.reset = latest
```

Consumer starts from **new incoming messages only**.

---

### Step 3 — Consumer Sends Fetch Requests

Internally the consumer sends requests like:

```
FetchRequest(topic="orders", partition=0, offset=5)
```

Broker returns messages starting from that offset.

---

# 6️⃣ Order in Which Messages Are Consumed

Kafka guarantees ordering **inside a partition only**.

Example:

Partition 0:

```
offset0 → msg1
offset1 → msg4
offset2 → msg7
offset3 → msg10
```

Consumer reads **strictly in this order**:

```
msg1
msg4
msg7
msg10
```

However, **across partitions there is no global order**.

Example:

```
partition0 → msg1 msg4 msg7
partition1 → msg2 msg5 msg8
partition2 → msg3 msg6 msg9
```

Processing order may be:

```
msg1
msg3
msg2
msg4
msg6
msg5
```

Order is guaranteed **per partition only**.

---

# 7️⃣ How Consumer Notifies Broker (Offset Commit Clarified)

Kafka **does not track message processing success**.

Instead, it tracks **offset commits**.

After processing messages, the consumer **commits the next offset**.

Example partition:

```
offset0 → msg1
offset1 → msg4
offset2 → msg7
offset3 → msg10
```

Consumer processes:

```
msg1
msg4
msg7
```

Then consumer commits:

```
offset = 3
```

Meaning:

```
"Everything before offset 3 is processed"
```

---

### What Happens Internally

Consumer sends a request to the broker:

```
OffsetCommitRequest
```

Containing:

```
consumer_group = order-service
topic = orders
partition = 0
offset = 3
```

Broker writes this record into internal topic:

```
__consumer_offsets
```

Example stored entry:

```
(order-service, orders, partition0) → offset 3
```

---

# 8️⃣ Two Ways Offset Commit Happens

---

### Auto Commit

```
enable.auto.commit = true
```

Consumer automatically commits offsets every few seconds.

Problem:

```
processing fails after commit
→ message lost
```

---

### Manual Commit (Recommended)

After processing:

```python
consumer.commit()
```

This sends the commit request explicitly.

---

# 9️⃣ What Broker Does After Consumer Commits

Broker simply:

1. Stores committed offset
2. Updates consumer group metadata

Important:

Kafka **does not delete the consumed message**.

Messages stay in the log.

Deletion happens only via **retention policy**.

Example:

```
log.retention.hours = 168
```

Messages older than 7 days are removed.

---

# 🔟 What Happens If Consumer Crashes (Clarified)

Example partition:

```
offset0 → msg1
offset1 → msg4
offset2 → msg7
offset3 → msg10
```

Scenario:

Consumer processed:

```
msg1
msg4
msg7
```

But committed only:

```
offset = 2
```

Then the consumer crashes.

Broker still stores:

```
last_committed_offset = 2
```

When a new consumer starts with the same group:

Kafka resumes from:

```
offset 2
```

So message:

```
msg7
```

will be **delivered again**.

This results in **duplicate processing**, but **no data loss**.

This is Kafka's **at-least-once delivery guarantee**.

# ✅ Final Simple Mental Model

```
Producer → sends messages
        ↓
Broker → stores messages in partition logs
        ↓
Consumer → assigned partitions automatically
        ↓
Consumer fetches messages from offset
        ↓
Consumer processes messages
        ↓
Consumer commits offset
        ↓
Broker stores offset in __consumer_offsets
```

Messages remain until **retention policy deletes them**.
