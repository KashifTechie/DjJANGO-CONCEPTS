🏭 Producer

Creates events, serializes them to bytes, and pushes into Kafka. Never writes directly — uses an internal buffer + batch manager for efficiency.

⚡ Broker

Stores messages in an append-only log, organized by topic → partition → offset. Replicates across nodes. Never deletes on consume.

🔄 Consumer

Reads from a partition at a specific offset. Commits the offset after processing. Part of a consumer group — each partition assigned to exactly one consumer.

Idempotence — Preventing Duplicates on Retry

Without idempotence: network failure after broker writes but before ACK → producer retries → duplicate message. With idempotence, the broker detects and discards it.

🔁 retry without vs with idempotence
❌ WITHOUT IDEMPOTENCE

2
Broker writes msg#42 to log ✓
3

4
Producer retries → sends msg#42 again

Broker writes msg#42 again → 💀 duplicate!
✅ WITH IDEMPOTENCE

Delivery Callback — delivery_report()
delivery callback Python 
def delivery_report(err, msg):
    # err  → KafkaError or None
    # msg  → Message object (topic, partition, offset, key, value)

    if err is not None:
        # Delivery failed after all retries + timeout
        # Possible errors: MSG_TIMED_OUT, UNKNOWN_TOPIC, etc.
        print("Delivery failed:", err)

    else:
        # Success — message is now durable in Kafka
        print(
            "Delivered to", msg.topic(),        # "orders"
            "partition",  msg.partition(),    # 0, 1, or 2
            "offset",     msg.offset()         # monotonically increasing
        )

⏰ When does this fire?

Only when you call producer.poll() or producer.flush(). Kafka Python is single-threaded from the callback perspective — it queues internal events and drains them on poll.

📍 msg.offset()

The position in the partition log where your message landed. Monotonically increasing. Use this for tracing, debugging, and building exactly-once consumer logic.

🔢 msg.partition()

Determined by hash(key) % num_partitions. Same key always lands in same partition — guarantees ordering for a given entity (e.g. all events for order_id).

🔑
Ordering guarantee: Within a single partition, messages are strictly ordered by offset. If you use order_id as the key, all events for the same order land in the same partition and will be consumed in order.
⚠️
No ordering across partitions. If two events are in different partitions (different keys), Kafka gives no ordering guarantee between them. Design your key strategy carefully.

✅ enable.auto.commit = True

Kafka auto-commits the highest consumed offset every 5000ms. Simple but risk: if the consumer crashes between processing offset 5 and the next commit, offsets 5–6 will be re-read on restart. At-least-once delivery.

🎯 Manual commit (production)

Set enable.auto.commit=False and call consumer.commit() after successfully processing. Ensures you only commit what you actually handled. Prevents data loss on crash. Exactly-once when combined with idempotent consumers.

🔴
auto.offset.reset = "earliest" — If this consumer group has never committed an offset (first startup), start reading from the beginning of the topic. Change to "latest" to skip all historical messages and only process new ones.

💓 what happens inside consumer.poll(1.0)?

Proves consumer is alive. If heartbeat gap exceeds session.timeout.ms (10s), group coordinator assumes dead → triggers rebalance.

Doesn't wait for the 1.0s timeout. Returns as soon as a message is in the local fetch buffer.

The 1.0 is a maximum wait. Allows the loop to stay responsive and still send heartbeats during quiet periods.

Not an exception — you must explicitly check if msg.error(). Log and continue, or implement dead-letter logic.

💡
heartbeat.interval.ms = 3000 vs session.timeout.ms = 10000. Heartbeat fires every 3s. Session times out at 10s. This gives ~3 missed heartbeats before the broker considers the consumer dead. Rule of thumb: heartbeat ≈ session_timeout / 3.

Graceful Shutdown — Signal Handling

signal.SIGINT

Sent when user presses Ctrl+C. Without a handler, Python raises KeyboardInterrupt mid-loop — leaving unflushed messages and uncommitted offsets.

signal.SIGTERM

Sent by docker stop, systemd, kubernetes pod termination. Without a handler, process is forcibly killed after a grace period. The handler gives you time to flush/close cleanly.

🛑 graceful shutdown sequence
SIGINT / SIGTERM
OS signal
shutdown(sig, frame) fires

Sets running = False. Does NOT interrupt the current loop body. Lets the current iteration complete naturally.

while loop checks
next iteration
while running: → exits loop

Loop condition is now False. Execution falls through to cleanup code below the loop.

producer path
after loop
producer.flush() — blocks until all delivered

No messages left in buffer. All delivery callbacks fired. Safe to exit.

consumer path
after loop
consumer.close() — commits offsets + leaves group

Sends LeaveGroup request. Group coordinator immediately rebalances remaining consumers. No need to wait for session timeout.

producer.py — Annotated Full Code
producer.py Python 
# ─── IMPORTS ────────────────────────────────────────────
from confluent_kafka import Producer  # handles: connect, buffer, batch, retry
import json                              # dict → JSON string
import uuid                              # generate unique order IDs
import time                              # time.sleep() to pace events
import signal                            # catch Ctrl+C / docker stop
import sys                               # sys.exit() for clean termination

# ─── CONFIGURATION ──────────────────────────────────────
conf = {
    "bootstrap.servers":  "localhost:9092",  # initial contact point for cluster metadata
    "client.id":          "order-service-producer",  # shows in broker logs / metrics

    "acks":               "all",    # wait for leader + all ISR replicas
    "enable.idempotence": True,    # broker dedupes retried batches (requires acks=all)

    "retries":            10,      # retry transient failures up to 10 times
    "retry.backoff.ms":   1000,   # wait 1s between retries

    "linger.ms":          20,      # wait 20ms before sending (batching window)
    "batch.size":         65536,  # max batch = 64KB; sends immediately when full

    "compression.type":   "snappy",  # compress batch before network send
    "message.timeout.ms": 30000     # total time incl. retries; fails after 30s
}

producer = Producer(conf)  # starts network thread + buffer + batch manager
topic = "orders"

# ─── DELIVERY CALLBACK ──────────────────────────────────
def delivery_report(err, msg):
    # fires on producer.poll() or flush() after broker responds
    if err is not None:
        print("Delivery failed:", err)      # log, alert, dead-letter
    else:
        print("Delivered to", msg.topic(), "partition", msg.partition(), "offset", msg.offset())

# ─── GRACEFUL SHUTDOWN ──────────────────────────────────
running = True

def shutdown(sig, frame):
    global running
    running = False  # lets current loop body finish, then exits

signal.signal(signal.SIGINT,  shutdown)  # Ctrl+C
signal.signal(signal.SIGTERM, shutdown)  # docker stop / kill

# ─── EVENT LOOP ─────────────────────────────────────────
while running:
    event = {
        "order_id": str(uuid.uuid4()),  # unique → deterministic partition assignment
        "user_id":  "user123",
        "amount":   250,
        "status":   "created"
    }

    message = json.dumps(event)  # serialize: dict → JSON string

    producer.produce(
        topic=topic,
        value=message.encode("utf-8"),   # Kafka messages are bytes, not strings
        key=event["order_id"],            # key controls which partition
        callback=delivery_report           # register callback for this message
    )

    producer.poll(0)    # non-blocking: drains internal event queue (fires pending callbacks)
    time.sleep(2)       # simulate new event every 2s

# ─── SHUTDOWN CLEANUP ───────────────────────────────────
print("Flushing pending messages...")
producer.flush()     # BLOCKS until all buffered messages delivered — no data loss
sys.exit(0)

consumer.py — Annotated Full Code
consumer.py Python 
# ─── IMPORTS ────────────────────────────────────────────
from confluent_kafka import Consumer   # handles: fetch, offset tracking, heartbeats
import json
import signal
import sys

# ─── CONFIGURATION ──────────────────────────────────────
conf = {
    "bootstrap.servers":       "localhost:9092",

    "group.id":                "order-processing-service",
    # group.id groups multiple consumer instances to share partitions
    # each partition assigned to exactly ONE consumer in the group

    "enable.auto.commit":      True,     # auto-commit offset every 5s
    "auto.commit.interval.ms": 5000,    # 5s commit interval

    "auto.offset.reset":       "earliest",  # on first start: read from beginning

    "session.timeout.ms":      10000,   # 10s: after this without heartbeat → dead
    "heartbeat.interval.ms":   3000,    # send heartbeat every 3s (≈ timeout/3)

    "max.poll.interval.ms":    300000   # 5 min: max time between poll() calls
                                          # increase if processing takes a long time
}

consumer = Consumer(conf)
topic = "orders"

consumer.subscribe([topic])  # joins consumer group, triggers partition assignment

# ─── GRACEFUL SHUTDOWN ──────────────────────────────────
running = True

def shutdown(sig, frame):
    global running
    running = False

signal.signal(signal.SIGINT,  shutdown)
signal.signal(signal.SIGTERM, shutdown)

# ─── POLL LOOP ──────────────────────────────────────────
while running:

    msg = consumer.poll(1.0)
    # poll(1.0): block up to 1s waiting for a message
    # also sends heartbeats and handles rebalance internally

    if msg is None:
        continue           # no message yet — loop and poll again

    if msg.error():
        print("Consumer error:", msg.error())
        continue           # not an exception — check explicitly and handle

    event = json.loads(msg.value().decode("utf-8"))  # bytes → string → dict

    print(
        "Received event:",
        "topic:",     msg.topic(),
        "partition:", msg.partition(),  # which partition this came from
        "offset:",    msg.offset(),     # position in the partition log
        "data:",      event
    )
    # ← your business logic here (save to DB, call API, etc.)

# ─── SHUTDOWN CLEANUP ───────────────────────────────────
print("Closing consumer...")
consumer.close()
# close() does two things:
# 1. commits any pending offsets (if auto.commit=True)
# 2. sends LeaveGroup → immediate rebalance (vs waiting 10s for timeout)

sys.exit(0)

Error Handling — What Can Go Wrong

❌ MSG_TIMED_OUT

Producer: message exceeded message.timeout.ms (30s) without delivery. All retries exhausted. Log it, alert, consider dead-letter queue. Do not silently ignore.

❌ UNKNOWN_TOPIC_OR_PART

Topic doesn't exist and auto.create.topics.enable=false on the broker. Create the topic before producing, or enable auto-create in dev.

⚠ OFFSET_OUT_OF_RANGE

Consumer trying to read an offset that no longer exists (log was compacted/deleted). Happens when consumer is behind the retention window. Handle with auto.offset.reset.

⚠ REBALANCE_IN_PROGRESS

Consumer group is reassigning partitions. During this time, commits may fail. Implement on_assign and on_revoke callbacks in production to handle gracefully.
