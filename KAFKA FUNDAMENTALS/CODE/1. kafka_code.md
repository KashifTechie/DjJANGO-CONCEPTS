## 0. System Architecture

### The Big Picture

```
┌─────────────────────┐        produce()         ┌──────────────────────────┐        poll()         ┌──────────────────────────┐
│      PRODUCER       │  ──── key+value+topic ──► │      KAFKA BROKER        │  ── offset tracking ──► │       CONSUMER           │
│   order-service     │                           │      localhost:9092       │                        │  order-processing-svc    │
│                     │                           │                           │                        │                          │
│ serialize           │                           │  topic: orders            │                        │ deserialize              │
│   → buffer          │                           │  ├─ partition 0           │                        │   → process              │
│     → batch         │                           │  ├─ partition 1           │                        │     → commit offset      │
│       → send        │                           │  └─ partition 2           │                        │                          │
└─────────────────────┘                           └──────────────────────────┘                        └──────────────────────────┘
```

### Three Roles — One Sentence Each

| Role | Responsibility |
|------|---------------|
| **Producer** | Creates events, serializes to bytes, pushes into Kafka via internal buffer + batch manager |
| **Broker** | Stores messages in an append-only log (topic → partition → offset). Replicates. Never deletes on consume |
| **Consumer** | Reads a partition at a specific offset. Commits offset after processing. Part of a group — each partition assigned to exactly one consumer |

## WHEN DIAGRAMS

### Producer — What Triggers What

```
TRIGGER                          →   WHAT FIRES
─────────────────────────────────────────────────────────────────────────────────
linger.ms expires (20ms)         →   batch.send()
                                     Flushes batch even if not full.
                                     Prevents messages sitting idle.

batch.size reached (64KB)        →   batch.send() immediately
                                     Ignores remaining linger.ms time.
                                     Throughput > latency priority here.

broker sends ACK (acks=all)      →   delivery_report(err=None, msg)
                                     Callback fires on thread calling poll().
                                     Prints partition + offset of confirmed write.

broker fails (network/timeout)   →   retry(backoff=1s) × 10
                                     Auto-retried up to 10 times with 1s gap.
                                     Idempotence ensures no duplicates.

message.timeout.ms exceeded      →   delivery_report(err=MSG_TIMED_OUT)
(30s elapsed)                        Message is dropped after all retries.
                                     Must handle: log / dead-letter / alert.

SIGINT / SIGTERM received        →   running = False → flush()
(Ctrl+C / docker stop)               Loop exits cleanly.
                                     flush() blocks until all buffered msgs delivered.
                                     Zero data loss on shutdown.
```

---

### Consumer — What Triggers What

```
TRIGGER                          →   WHAT FIRES
─────────────────────────────────────────────────────────────────────────────────
poll(1.0) called                 →   1. Sends heartbeat to group coordinator
(every loop tick)                    Proves consumer is alive.
                                     If gap > session.timeout.ms → rebalance.

message available in broker      →   poll() returns Message immediately
                                     Does not wait the full 1.0s timeout.
                                     Returns as soon as message is in local buffer.

no messages (empty partition)    →   poll() returns None after 1.0s
                                     Allows loop to stay responsive
                                     and still send heartbeats during quiet periods.

broker error (KafkaError)        →   poll() returns Message with msg.error() set
                                     NOT an exception — must check msg.error() explicitly.
                                     Log and continue, or dead-letter.

heartbeat gap > session.timeout  →   group rebalance triggered
(10s without heartbeat)              Consumer considered dead.
                                     Partitions reassigned to other consumers in group.

poll() gap > max.poll.interval   →   consumer kicked from group
(300s without poll)                  Distinct from heartbeat timeout.
                                     Means consumer is alive but too slow.
                                     Increase if processing is legitimately slow.

SIGINT / SIGTERM received        →   running = False → consumer.close()
(Ctrl+C / docker stop)               close() commits pending offsets.
                                     Sends LeaveGroup → immediate rebalance.
                                     No need to wait for 10s session timeout.
```

## 1. Producer — Configuration

```python
conf = {
    "bootstrap.servers":  "localhost:9092",
    "client.id":          "order-service-producer",
    "acks":               "all",
    "enable.idempotence": True,
    "retries":            10,
    "retry.backoff.ms":   1000,
    "linger.ms":          20,
    "batch.size":         65536,
    "compression.type":   "snappy",
    "message.timeout.ms": 30000
}
```

### Parameter Reference

| Parameter | Value | What it controls | Impact area |
|-----------|-------|-----------------|-------------|
| `bootstrap.servers` | `localhost:9092` | Initial broker for cluster metadata discovery only — not the only server msgs go to | Connection |
| `acks` | `"all"` | How many replicas must confirm before ACK. `"all"` waits for all in-sync replicas — safest, adds ~5–20ms latency | **Durability** |
| `enable.idempotence` | `True` | Assigns `producer_id + sequence_number` per batch. Broker deduplicates retried batches. Requires `acks=all` | **Exactly-once** |
| `retries` | `10` | Auto-retry count on transient failures. Safe with idempotence — retries won't produce duplicates | Reliability |
| `retry.backoff.ms` | `1000` | Milliseconds to wait between retries. Prevents thundering herd on broker recovery | Reliability |
| `linger.ms` | `20` | Max wait before sending an incomplete batch. Trades latency for throughput — batches more messages together | **Throughput** |
| `batch.size` | `65536` | 64 KB max batch per partition. Sends when full OR linger.ms expires — whichever is first | **Throughput** |
| `compression.type` | `"snappy"` | Compresses entire batch before TCP send. Snappy = fast, ~2x ratio. Good CPU/savings balance | Bandwidth |
| `message.timeout.ms` | `30000` | Total delivery time including all retries. After 30s, delivery fails and callback fires with error | Safety |

## 2. Producer — Message Lifecycle

```
produce()        buffer         compress       broker         callback
   │               │               │              │              │
   ▼               ▼               ▼              ▼              ▼
┌──────┐  enqueue  ┌──────┐ batch  ┌──────┐ TCP  ┌──────┐ ACK  ┌──────┐
│ app  │ ────────► │in-mem│ full / │snappy│ send │write │ ───► │deliv │
│ code │           │buffer│ linger │ gzip │      │to log│      │_reprt│
└──────┘           └──────┘        └──────┘      └──────┘      └──────┘
```

### Key Insight — `produce()` is non-blocking

- `produce()` puts the message in an internal buffer and **returns instantly**
- The actual network send happens on a **background thread**
- `poll(0)` drains the internal event queue and fires pending callbacks **on your thread**
- `flush()` blocks until the buffer is empty — use it on shutdown

> ⚠️ **Never skip `flush()` on shutdown.** If the process exits before the buffer is drained, buffered messages are silently lost.

## 3. Batching — linger.ms vs batch.size

### Two independent send triggers — whichever fires first wins

**`linger.ms = 20`** — the latency knob
- Producer waits up to 20ms before sending, hoping more messages arrive
- Set higher → more throughput (larger batches)
- Set lower → less latency (smaller batches, more sends)

**`batch.size = 65536`** — the throughput ceiling
- Max bytes per batch per partition (64 KB)
- When full, sends immediately — ignores remaining linger time

### Batch fill diagram

```
Scenario A — linger.ms fires (low traffic)
──────────────────────────────────────────
t=0ms   [██░░░░░░░░]  2 msgs  ~5KB
t=8ms   [████░░░░░░]  7 msgs  ~18KB
t=20ms  [██████░░░░]  12 msgs ~30KB
            ⏰ linger.ms fires → sends half-full 30KB batch

Scenario B — batch.size fires (burst traffic)
─────────────────────────────────────────────
t=0ms   [███░░░░░░░]  5 msgs
t=5ms   [██████░░░░]  burst arrives
t=12ms  [██████████]  64KB FULL
            🚀 size fires early → sends immediately, ignores linger
```

## 4. Acknowledgements — acks=0, 1, all

```
acks=0          acks=1              acks=all
───────         ──────────          ────────────────────────────
Producer        Producer            Producer
   │               │                   │
   ▼               ▼                   ▼
 send()          send()             send()
   │               │                   │
   │            Leader                Leader
   │            writes ✓             writes ✓
   │               │                   │
   │            ACK sent            Replica 1
   │               │                writes ✓
   X          (replicas not           │
(no ack)       waited for)         Replica 2
                                   writes ✓
                                       │
                                    ACK sent

Risk:          Risk:               Risk:
can lose       lose if leader      ~5-20ms added latency
any message    fails before        (safest — production
               replication         recommended)
```

| Setting | Durability | Latency | Use case |
|---------|-----------|---------|----------|
| `acks=0` | ❌ None | Lowest | Metrics, logs you can afford to lose |
| `acks=1` | ⚠️ Partial | Medium | Moderate risk tolerance |
| `acks=all` | ✅ Maximum | +5–20ms | Financial, orders, critical events |

> **ISR = In-Sync Replicas.** Replicas fully caught up with the leader. `acks=all` only waits for ISR members — doesn't block on slow replicas.

## 5. Idempotence — Preventing Duplicates on Retry

Without idempotence: network failure after broker writes but before ACK → producer retries → **duplicate message**.

### Without idempotence ❌

```
Step 1: Producer sends msg#42
Step 2: Broker writes msg#42 to log ✓
Step 3: Network drops → producer never receives ACK ✗
Step 4: Producer retries → sends msg#42 again
Step 5: Broker writes msg#42 AGAIN → 💀 DUPLICATE
```

### With idempotence ✅

```
Step 1: Producer assigned pid=7, seq=42
Step 2: Broker writes batch, records (pid=7, seq=42) ✓
Step 3: Network drops → no ACK received ✗
Step 4: Producer retries with same pid=7, seq=42
Step 5: Broker sees duplicate seq → silently discards ✓
```

### How it works internally

```
Each batch gets stamped:
┌──────────────────────────────────┐
│  producer_id  = 7                │  ← unique per producer session
│  sequence_num = 42               │  ← monotonically increasing per partition
│  payload      = { order: ... }   │
└──────────────────────────────────┘

Broker tracks highest seen sequence per (producer_id, partition).
If seq ≤ last_seen → duplicate → drop.
If seq = last_seen + 1 → new → accept.
```

> Requires `acks=all`. Automatically enabled when `enable.idempotence=True`.

## 6. Delivery Callback

```python
def delivery_report(err, msg):
    # fires on producer.poll() or flush() after broker responds
    if err is not None:
        print("Delivery failed:", err)      # log, alert, dead-letter
    else:
        print(
            "Delivered to", msg.topic(),     # "orders"
            "partition",    msg.partition(), # 0, 1, or 2
            "offset",       msg.offset()    # monotonically increasing
        )
```

### Three things to know

**When does it fire?**
- Only when you call `producer.poll()` or `producer.flush()`
- Kafka Python queues internal events and drains them on poll
- It does NOT fire automatically on a background thread

**`msg.offset()`**
- The position in the partition log where your message landed
- Monotonically increasing per partition
- Use for tracing, debugging, and building exactly-once consumer logic

**`msg.partition()`**
- Determined by `hash(key) % num_partitions`
- Same key always → same partition → ordering guaranteed for that key

## 7. Partitions — Key-based Routing

### How keys map to partitions

```
key = order_id (e.g. "uuid-abc")

hash("uuid-abc") % 3 = 1  →  goes to partition 1
hash("uuid-xyz") % 3 = 0  →  goes to partition 0
hash("uuid-def") % 3 = 2  →  goes to partition 2
```

### Topic "orders" — 3 partitions

```
PARTITION 0          PARTITION 1          PARTITION 2
────────────         ────────────         ────────────
offset | key         offset | key         offset | key
   0   │ user_a         0   │ user_b         0   │ user_c
   1   │ user_a         1   │ user_b         1   │ user_e
   2   │ user_d                              2   │ user_c

All events for user_a → always partition 0 → strict ordering ✓
All events for user_b → always partition 1 → strict ordering ✓
```

### Two critical rules

> ✅ **Ordering within a partition is guaranteed.** All events for the same `order_id` land in the same partition and will be consumed in offset order.

> ⚠️ **No ordering across partitions.** If two events are in different partitions, Kafka gives zero ordering guarantee between them. Design your key strategy carefully.

## 8. Consumer — Configuration

```python
conf = {
    "bootstrap.servers":       "localhost:9092",
    "group.id":                "order-processing-service",
    "enable.auto.commit":      True,
    "auto.commit.interval.ms": 5000,
    "auto.offset.reset":       "earliest",
    "session.timeout.ms":      10000,
    "heartbeat.interval.ms":   3000,
    "max.poll.interval.ms":    300000
}
```

### Parameter Reference

| Parameter | Value | What it controls |
|-----------|-------|-----------------|
| `group.id` | `"order-processing-service"` | Groups multiple consumer instances to share partitions. Each partition assigned to exactly **one** consumer in the group |
| `enable.auto.commit` | `True` | Auto-commits offset every 5s. Simple but risk: crash between processing and commit → messages re-read on restart |
| `auto.commit.interval.ms` | `5000` | How often the offset is persisted to `__consumer_offsets` topic |
| `auto.offset.reset` | `"earliest"` | On first startup (no committed offset): read from the **beginning**. Set `"latest"` to skip history |
| `session.timeout.ms` | `10000` | If no heartbeat received for 10s, coordinator declares consumer dead → rebalance |
| `heartbeat.interval.ms` | `3000` | Send heartbeat every 3s. Rule of thumb: `heartbeat ≈ session_timeout / 3` |
| `max.poll.interval.ms` | `300000` | Max time between `poll()` calls. If exceeded, consumer is kicked from group. Increase if processing is slow |

## 9. Offset Management

### What "offset" means

```
Partition 0 — offset timeline:

[0]✅ [1]✅ [2]✅ [3]✅ [4]👁️ [5]🔄 [6]🔄 [7]💤 [8]💤
 │committed│              │current │pending │  unread  │

✅ committed  — offset persisted to __consumer_offsets
👁️ current    — currently being processed
🔄 pending    — read but not yet committed (window of re-read risk)
💤 unread     — not yet fetched from broker
```

### Auto-commit vs Manual commit

**Auto-commit (`enable.auto.commit=True`)**
- Kafka auto-commits the highest consumed offset every 5000ms
- Simple, zero code required
- Risk: crash between processing offset 5 and next commit → offsets 5–6 re-read on restart
- Delivery guarantee: **at-least-once**

**Manual commit (`enable.auto.commit=False`) — production recommended**
```python
# call after successfully processing the message
consumer.commit(msg)
```
- Only commits what you actually handled
- Prevents data loss on crash
- Combined with idempotent consumers → **exactly-once**

> 🔴 `auto.offset.reset = "earliest"` — If consumer group has no committed offset (first startup), start from the beginning of the topic. Set to `"latest"` to skip all historical messages.

## 10. Poll Loop — Heartbeats & Rebalance

### What happens inside `consumer.poll(1.0)`

```
poll(1.0) called
      │
      ├─► 1. Send heartbeat to group coordinator
      │         Proves consumer is alive.
      │         gap > session.timeout.ms → rebalance triggered.
      │
      ├─► 2. Check local fetch buffer
      │         Message ready?  → return Message immediately (no wait)
      │         No message?     → wait up to 1.0s
      │         Timeout elapsed → return None
      │
      └─► 3. Handle broker errors
                Return Message with msg.error() set.
                NOT an exception — check explicitly.
```

### Heartbeat vs Poll timeout — they're different

```
heartbeat.interval.ms = 3000    ← "I'm alive" signal
session.timeout.ms    = 10000   ← deadline before declared dead

3 missed heartbeats → declared dead → rebalance

max.poll.interval.ms = 300000   ← "I'm still processing" deadline
                                   Consumer alive but too slow to poll
                                   → kicked from group
```

> 💡 **Rule:** `heartbeat.interval.ms ≈ session.timeout.ms / 3`
> This gives ~3 missed heartbeats as buffer before the broker considers the consumer dead.

## 11. Graceful Shutdown

### Why signal handlers matter

| Signal | Sent by | Without handler |
|--------|---------|-----------------|
| `SIGINT` | Ctrl+C | Python raises `KeyboardInterrupt` mid-loop — unflushed messages, uncommitted offsets |
| `SIGTERM` | docker stop, systemd, k8s | Process forcibly killed — same risk |

### Shutdown sequence

```
SIGINT / SIGTERM received
        │
        ▼
shutdown(sig, frame) fires
  └─► running = False
        (does NOT interrupt current loop body — lets it finish)
        │
        ▼
while running: → condition False → exits loop
        │
        ├─► PRODUCER PATH
        │     producer.flush()
        │     blocks until all buffered messages delivered
        │     all delivery callbacks fired
        │     safe to exit ✓
        │
        └─► CONSUMER PATH
              consumer.close()
              commits pending offsets
              sends LeaveGroup request
              group coordinator rebalances immediately
              (vs waiting 10s for session timeout) ✓
```

## 12. Error Handling

### Common errors and responses

| Error | Side | Cause | Response |
|-------|------|-------|----------|
| `MSG_TIMED_OUT` | Producer | Message exceeded `message.timeout.ms` after all retries | Log + alert + dead-letter queue |
| `UNKNOWN_TOPIC_OR_PART` | Producer | Topic doesn't exist; `auto.create.topics.enable=false` | Create topic first, or enable auto-create in dev |
| `OFFSET_OUT_OF_RANGE` | Consumer | Trying to read offset that no longer exists (log compacted/deleted) | Handle with `auto.offset.reset` |
| `REBALANCE_IN_PROGRESS` | Consumer | Group is reassigning partitions | Implement `on_assign` / `on_revoke` callbacks |

### Error check pattern (consumer)

```python
msg = consumer.poll(1.0)

if msg is None:
    continue                    # no message — normal

if msg.error():
    print("Error:", msg.error())
    continue                    # not an exception — must check explicitly

# safe to process msg here
event = json.loads(msg.value().decode("utf-8"))
```

## P. producer.py — Annotated Full Code

```python
# ─── IMPORTS ────────────────────────────────────────────────────────────────
from confluent_kafka import Producer  # handles: connect, buffer, batch, retry
import json                           # dict → JSON string
import uuid                           # generate unique order IDs
import time                           # time.sleep() to pace events
import signal                         # catch Ctrl+C / docker stop
import sys                            # sys.exit() for clean termination

# ─── CONFIGURATION ──────────────────────────────────────────────────────────
conf = {
    "bootstrap.servers":  "localhost:9092",  # initial contact point for cluster metadata
    "client.id":          "order-service-producer",  # shows in broker logs / metrics

    "acks":               "all",    # wait for leader + all ISR replicas
    "enable.idempotence": True,     # broker dedupes retried batches (requires acks=all)

    "retries":            10,       # retry transient failures up to 10 times
    "retry.backoff.ms":   1000,     # wait 1s between retries

    "linger.ms":          20,       # wait 20ms before sending (batching window)
    "batch.size":         65536,    # max batch = 64KB; sends immediately when full

    "compression.type":   "snappy", # compress batch before network send
    "message.timeout.ms": 30000     # total time incl. retries; fails after 30s
}

producer = Producer(conf)  # starts network thread + buffer + batch manager
topic = "orders"

# ─── DELIVERY CALLBACK ──────────────────────────────────────────────────────
def delivery_report(err, msg):
    # fires on producer.poll() or flush() after broker responds
    if err is not None:
        print("Delivery failed:", err)      # log, alert, dead-letter
    else:
        print(
            "Delivered to", msg.topic(),    # "orders"
            "partition",    msg.partition(),# 0, 1, or 2
            "offset",       msg.offset()   # monotonically increasing
        )

# ─── GRACEFUL SHUTDOWN ──────────────────────────────────────────────────────
running = True

def shutdown(sig, frame):
    global running
    running = False  # lets current loop body finish, then exits

signal.signal(signal.SIGINT,  shutdown)  # Ctrl+C
signal.signal(signal.SIGTERM, shutdown)  # docker stop / kill

# ─── EVENT LOOP ─────────────────────────────────────────────────────────────
while running:
    event = {
        "order_id": str(uuid.uuid4()),  # unique → deterministic partition assignment
        "user_id":  "user123",
        "amount":   250,
        "status":   "created"
    }

    message = json.dumps(event)  # serialize: dict → JSON string

    producer.produce(
        topic=topic,
        value=message.encode("utf-8"),  # Kafka messages are bytes, not strings
        key=event["order_id"],          # key controls which partition
        callback=delivery_report        # register callback for this message
    )

    producer.poll(0)   # non-blocking: drains internal event queue (fires pending callbacks)
    time.sleep(2)      # simulate new event every 2s

# ─── SHUTDOWN CLEANUP ───────────────────────────────────────────────────────
print("Flushing pending messages...")
producer.flush()  # BLOCKS until all buffered messages delivered — no data loss
sys.exit(0)
```

## C. consumer.py — Annotated Full Code

```python
# ─── IMPORTS ────────────────────────────────────────────────────────────────
from confluent_kafka import Consumer  # handles: fetch, offset tracking, heartbeats
import json
import signal
import sys

# ─── CONFIGURATION ──────────────────────────────────────────────────────────
conf = {
    "bootstrap.servers":       "localhost:9092",

    "group.id":                "order-processing-service",
    # group.id groups multiple consumer instances to share partitions
    # each partition assigned to exactly ONE consumer in the group

    "enable.auto.commit":      True,    # auto-commit offset every 5s
    "auto.commit.interval.ms": 5000,   # 5s commit interval

    "auto.offset.reset":       "earliest",  # on first start: read from beginning

    "session.timeout.ms":      10000,  # 10s: after this without heartbeat → dead
    "heartbeat.interval.ms":   3000,   # send heartbeat every 3s (≈ timeout/3)

    "max.poll.interval.ms":    300000  # 5 min: max time between poll() calls
                                       # increase if processing takes a long time
}

consumer = Consumer(conf)
topic = "orders"

consumer.subscribe([topic])  # joins consumer group, triggers partition assignment

# ─── GRACEFUL SHUTDOWN ──────────────────────────────────────────────────────
running = True

def shutdown(sig, frame):
    global running
    running = False

signal.signal(signal.SIGINT,  shutdown)
signal.signal(signal.SIGTERM, shutdown)

# ─── POLL LOOP ──────────────────────────────────────────────────────────────
while running:

    msg = consumer.poll(1.0)
    # poll(1.0): block up to 1s waiting for a message
    # also sends heartbeats and handles rebalance internally

    if msg is None:
        continue            # no message yet — loop and poll again

    if msg.error():
        print("Consumer error:", msg.error())
        continue            # not an exception — check explicitly and handle

    event = json.loads(msg.value().decode("utf-8"))  # bytes → string → dict

    print(
        "Received event:",
        "topic:",     msg.topic(),
        "partition:", msg.partition(),  # which partition this came from
        "offset:",    msg.offset(),     # position in the partition log
        "data:",      event
    )
    # ← your business logic here (save to DB, call API, etc.)

# ─── SHUTDOWN CLEANUP ───────────────────────────────────────────────────────
print("Closing consumer...")
consumer.close()
# close() does two things:
# 1. commits any pending offsets (if auto.commit=True)
# 2. sends LeaveGroup → immediate rebalance (vs waiting 10s for timeout)

sys.exit(0)
```

## Quick Reference Card

```
PRODUCER CHEATSHEET
───────────────────────────────────────────────────────
produce()          → non-blocking, adds to buffer
poll(0)            → drains callbacks (non-blocking)
flush()            → blocks until buffer empty (use on shutdown)
delivery_report()  → fires after ACK or final failure

acks=all           → safest, +latency
idempotence=True   → safe retries, no duplicates
linger.ms          → throughput/latency tradeoff
batch.size         → max bytes per batch per partition

CONSUMER CHEATSHEET
───────────────────────────────────────────────────────
subscribe([topic]) → join group, trigger partition assign
poll(1.0)          → fetch msg + send heartbeat (1.0s max wait)
msg.error()        → always check — not an exception
consumer.close()   → commit offsets + send LeaveGroup

session.timeout    → 10s: no heartbeat = declared dead
heartbeat.interval → 3s: send "I'm alive" signal
max.poll.interval  → 300s: too slow to poll = kicked

DELIVERY GUARANTEES
```
